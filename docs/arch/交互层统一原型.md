# 交互层统一模型

## 目标

本文件记录当前已经落地的交互层模型。未来 AI 如需继续扩展工具交互，应以这里为准，而不是恢复旧的 `targetMode`、`AimState` 或平铺 payload。

## 核心分层

- `shared`
  - 定义工具的 `interaction`
  - 定义 `ToolSelectionRecord`
  - 基于 `input` 计算 `PreviewDescriptor`
  - 基于 `input` 执行 authoritative `ActionResolution`
- `client`
  - 维护本地 `InteractionSession`
  - 解释鼠标、键盘和弹窗输入
  - 把当前输入状态转换为 `ToolSelectionRecord`
  - 调用 shared preview，并在最终确认时发送 `useTool`

## 工具输入契约

每个 `Tool` 在 shared 中都必须完整定义三件事：

- `definition`
- `interaction`
- `execute`

其中：

- `interaction`
  - 描述输入阶段和顺序
- `execute`
  - 描述规则效果
- 合法范围、目标筛选、预览字段
  - 由工具自己的规则逻辑决定
  - 不由 client 从交互模式反推

当前 `useTool` 的权威协议固定为：

```ts
interface UseToolCommandPayload {
  toolInstanceId: string;
  input: ToolSelectionRecord;
}
```

`ToolSelectionRecord` 是命名 slot 到选择值的映射。当前基础选择值类型只有三种：

- `direction`
- `tile`
- `choice`

## 交互阶段模型

client 侧统一维护：

```ts
interface InteractionSession {
  toolId: ToolId;
  toolInstanceId: string;
  interaction: ToolInteractionDefinition;
  stageIndex: number;
  committed: ToolSelectionRecord;
  draft: ToolSelectionRecord;
}
```

约束如下：

- `committed`
  - 已完成阶段的输入
- `draft`
  - 当前阶段的临时输入
- shared preview
  - 永远读取 `committed + draft`
- shared execute
  - 永远读取最终确认后的 `input`

## 当前基础阶段

当前 shared / client 都已支持以下基础阶段：

- `drag-direction-release`
  - 拖动选择方向，松开立即确认
- `drag-tile-release`
  - 拖动选择地块，松开立即确认
- `drag-axis-tile-release`
  - 在一次拖动中同时确定轴向方向和目标格
- `modal-choice`
  - 在弹窗中做离散选择
- `sequence([...])`
  - 把多个阶段顺序组合成多段工具交互

## 当前工具映射

- `movement / jump / hookshot / basketball / rocket / awmShoot`
  - `drag-direction-release`
- `buildWall / teleport`
  - `drag-tile-release`
- `brake`
  - `drag-axis-tile-release`
- `bombThrow`
  - `sequence([drag-tile-release, drag-direction-release])`
- `balance`
  - `modal-choice`

## 明确禁止

未来 AI 不应：

- 恢复 `targetMode` 或 `tileTargeting`
- 在 client 恢复 `AimState` 这种固定联合类型
- 向 `UseToolCommandPayload` 恢复平铺的 `direction / targetPosition / choiceId`
- 为单个工具在 room、simulator 或 client store 中新增专用输入字段
- 让 client 自己根据交互模式推导规则合法范围

如果需要新的复杂交互，正确做法是：

1. 在 shared 扩展通用 `Tool.interaction` 阶段模型
2. 在 client 扩展 `InteractionSession` 的解释方式
3. 仍然通过 `ToolSelectionRecord` 传递最终输入

## 验证要求

修改交互协议后，至少验证：

- `npm.cmd run typecheck`
- `npm.cmd run goldens`
- 浏览器烟测
  - 创建房间
  - 开局
  - 至少一条 `tile` 输入链路
  - 至少一条 `choice` 输入链路

当前已验证：

- `buildWall` 的 `tile` 输入可真实写入房间状态
- `balance` 的 `choice` 输入可真实写入玩家 tag
