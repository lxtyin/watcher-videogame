# 架构总览

## 目标

当前实现遵循需求文档中的三层拆分：

- `packages/shared`
  - 放置前后端共用的类型、常量、棋盘定义、掷骰逻辑、Tool 注册与动作结算函数。
- `packages/server`
  - 使用 Colyseus 维护权威房间状态，校验掷骰、使用 Tool 和结束回合。
- `packages/client`
  - 使用 React、Zustand 和 React Three Fiber 呈现 3D 原型，并把玩家意图发送给服务端。

## 当前运行链路

1. 客户端通过 `colyseus.js` 连接 `watcher_room`。
2. 服务端在玩家加入时创建棋子状态，并把首个玩家置为当前行动者。
3. 当前玩家在 `roll` 阶段发送 `rollDice`。
4. 服务端生成移动骰结果和工具骰结果，并把它们写入当前玩家的回合 `tools[]`。
   - 固定生成一个 `Movement(points)`
   - 再根据可配置工具骰面生成一个基础 Tool
5. 客户端在 `action` 阶段从头顶弧形 Tool 环里选择当前 Tool。
6. 若 Tool 为 `direction` Tool，客户端在 3D 场景内拖动方向，服务端调用共享层统一结算。
7. 若 Tool 为 `tile` Tool，客户端拖动并吸附目标格，再把目标格发给服务端统一结算。
8. 若 Tool 为即时 Tool，服务端同样调用共享层统一结算，并回写新的 `tools[]`。
9. Colyseus 将房间状态同步回客户端，客户端据此刷新 3D 场景、HUD 与事件日志。

## 当前已实现内容

- 7x7 原型棋盘
- `floor`、`wall`、`earthWall` 三种基础地形
- 新增地形扩展原型
  - `pit`
  - `lucky`
  - `conveyor`
- 双阶段回合流：`roll` -> `action`
- 双骰原型：移动骰子 + 工具骰子
- 工具骰面配置原型
  - 每个工具骰面都可以配置 Tool 类型与基础参数
- 统一的回合 Tool 列表模型
  - `Movement` 也是 Tool
  - Tool 通过实例 id 被选择和消耗
  - Tool 实例统一携带 `params`
  - Tool 可以派生新的 Tool
  - Tool 可以带使用条件
- 当前 Tool 原型
  - `Movement`
  - `Jump`
  - `Hookshot`
  - `Dash`
  - `Brake`
  - `BuildWall`
  - `Basketball`
  - `Rocket`
  - `Teleport`
- 当前 Tool 组合逻辑
  - `Dash` 为当前列表中的所有 `Movement` +2
  - `Dash` 只有在列表中存在 `Movement` 时才可用
  - `Brake(3)` 会朝目标轴向移动至多 3 格，并停在实际可达的最远格
  - `BuildWall` 会把周围八格中的空地改造成土墙
  - `Basketball` 会沿方向飞行、遇墙反弹、命中玩家后推动并返还篮球次数
  - `Rocket` 会在碰撞点触发爆炸，并对中心与周围玩家施加不同位移
  - `Teleport` 会直接跳到任意可落脚地块
- 共享地形结算链路
  - `actions.ts` 先结算 Tool 的直接效果，再统一追加停留地形效果
  - `terrain.ts` 用注册表集中定义“经过”和“停留”地形行为
  - `Movement` 与 `Brake` 复用同一条地面遍历逻辑
  - `Jump` 仍走独立飞跃逻辑，不触发加速带经过效果
- 基于共享规则的客户端路径预览
- 3D 场景内执行动作
  - 头顶弧形 Tool 环动态展示本回合 `tools[]`
  - Tool 条件不满足时按钮置灰
  - 定向 Tool 通过场景内 3D 箭头提示方向
  - 选格 Tool 通过不同的目标采样模式表达目标
    - `axis_line`
    - `adjacent_ring`
    - `board_any`
  - 交互流程为按下 Tool、拖动选目标、松手执行、右键取消
  - `Movement`、`Jump`、`Hookshot`、`Teleport` 会在预计落点或命中目标脚底下显示预览圈
  - `BuildWall` 会渲染目标土墙虚影
  - `Rocket` 会高亮预计爆炸十字区域
  - `Basketball` 只显示方向箭头，不额外高亮落点
- 调试入口
  - 左侧栏可在行动阶段直接发放任意已实现 Tool
  - `Teleport` 默认通过该调试入口获得
- 3D 地形表现原型
  - 坑洞、幸运方块与加速带都先用程序化几何表现
  - 加速带会在棋盘上直接显示方向箭头
- 固定倾斜俯视相机
- `window.render_game_to_text` 与 `window.advanceTime` 自动化桥接

## 动作表现补充

当前已经有一条独立于规则结算的表现链路：

- 共享层在 `ActionResolution` 之外返回 `ActionPresentation`
- 服务端把最近一次 `ActionPresentation` 同步给客户端
- 客户端按本地时间推进 `player_motion`、`projectile`、`effect`
- 客户端渲染时会保留角色上一稳定落位，直到对应 motion 真正开始

这样后续加入更多特效时，可以继续沿着“共享层产出语义事件、客户端决定具体视觉实现”的方向扩展，而不把动画细节重新塞回规则层。

## 2026-03-30 Refactor Update

当前目录结构在原型基础上进一步细化为：

```text
packages/
  client/
    src/
      game/
        animation/
        components/
        content/
        network/
        state/
        utils/
  server/
    src/
      rooms/
        WatcherRoom.ts
        roomEventLog.ts
        roomStateMappers.ts
        roomStateMutations.ts
      schema/
  shared/
    src/
      content/
      rules/
      actions.ts
      board.ts
      characters.ts
      effects.ts
      summons.ts
      tools.ts
      types.ts
```

这次重构的重点不是新增玩法，而是把“内容定义”和“运行时代码”分层：

- `shared/content`
  - 单点维护 Tool、角色、召唤物、表现事件、地图内容
- `shared/rules`
  - 按空间计算、Tool 执行、通用后处理、表现时间线拆分规则代码
- `server/rooms/*helper`
  - 把 Colyseus schema 与共享快照之间的映射和回写独立出来
- `client/game/content`
  - 把 UI token、模型 manifest 等前端内容配置抽离出组件

这样后续增加内容时，更接近“新增一条注册记录 + 在对应执行层补实现”，而不是继续放大单个 God file。
