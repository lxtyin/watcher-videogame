# 架构总览

## 三层结构

- `packages/shared`
  - 唯一权威规则层。
  - 保存共享类型、内容注册、回合编排、动作结算、移动系统、地形、召唤物、能力系统。
- `packages/server`
  - Colyseus authoritative room 适配层。
  - 负责房间生命周期、消息入口、schema 同步、重连和大厅流程。
- `packages/client`
  - React + Zustand + React Three Fiber 客户端。
  - 负责输入、预览、HUD、表现层播放和房间 UI。

## 核心数据流

1. server 用 shared 内容注册初始化房间状态。
2. client 只发送高层命令，例如 `rollDice`、`useTool`、`endTurn`。
3. `WatcherRoom` 把 schema 映射成 shared `GameSnapshot + GameRuntimeState`。
4. `WatcherRoom` 调用 `createGameOrchestrator(...).dispatch()` 或 `advanceTurn()`。
5. shared orchestration 完成回合推进、阶段切换、能力触发、动作结算、表现事件生成。
6. server 将 shared 结果回写到 Colyseus schema。
7. client 订阅 schema，反序列化为本地状态，再消费 shared `ActionPresentation` 做动画播放。

## 当前最重要的边界

### 回合编排边界

- 只有 `packages/shared/src/gameOrchestration.ts` 可以决定阶段推进、换人、结算和延迟推进。
- `WatcherRoom.ts` 和 `simulation/engine.ts` 不允许实现第二套 turn flow。

### 规则与表现边界

- shared 只产出语义结果：最终状态、事件日志、`ActionPresentation`。
- client 负责视觉表现、输入体验和预览，但不能重写真实规则。

### 内容与运行时边界

- 静态内容放在 `shared/content` 或 `client/game/content`。
- 真实行为放在 shared 规则层。
- `WatcherRoom` 和 `BoardScene` 不能再成为新规则的落点。
