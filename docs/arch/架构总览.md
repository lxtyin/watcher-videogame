# 架构总览

## 三层结构

- `packages/shared`
  - 权威规则层。
  - 保存共享类型、内容注册、底层移动系统、工具结算、地形与召唤物触发、角色运行时、回合编排。
- `packages/server`
  - Colyseus 权威房间层。
  - 负责房间生命周期、消息入口、schema 同步、重连与大厅流程。
- `packages/client`
  - React + Zustand + React Three Fiber 客户端。
  - 负责输入、预览、播放 shared presentation、HUD 与房间 UI。

## 当前关键入口

- shared
  - `packages/shared/src/gameOrchestration.ts`
    - 唯一的回合编排实现。
  - `packages/shared/src/actions.ts`
    - 工具动作统一入口。
  - `packages/shared/src/rules/movementSystem.ts`
    - 底层移动结算入口。
  - `packages/shared/src/terrain.ts`
    - 地形触发注册。
  - `packages/shared/src/summons.ts`
    - 召唤物触发注册。
  - `packages/shared/src/characterRuntime.ts`
    - 角色跨回合状态与回合资源运行时。
  - `packages/shared/src/content/`
    - 工具、角色、召唤物、地图、表现效果等内容注册。
- server
  - `packages/server/src/rooms/WatcherRoom.ts`
    - 房间消息入口与 orchestration 适配层。
  - `packages/server/src/rooms/roomStateMappers.ts`
    - schema -> shared snapshot 映射。
  - `packages/server/src/rooms/roomStateMutations.ts`
    - shared snapshot -> schema 回写。
  - `packages/server/src/schema/WatcherState.ts`
    - Colyseus 同步状态。
- client
  - `packages/client/src/game/network/useWatcherConnection.ts`
    - 房间连接与状态订阅。
  - `packages/client/src/game/utils/deserializeRoomState.ts`
    - schema 快照反序列化。
  - `packages/client/src/game/components/BoardScene.tsx`
    - 场景装配、输入与播放。
  - `packages/client/src/game/interaction/`
    - 瞄准与预览提取。
  - `packages/client/src/game/content/`
    - 客户端 UI 与检查卡内容。

## 核心数据流

1. 房间创建时，server 用 shared 地图定义初始化 `WatcherState`。
2. 客户端发送高层命令，例如 `rollDice`、`useTool`、`endTurn`。
3. `WatcherRoom` 将 schema 状态映射成 shared `GameSnapshot + GameRuntimeState`。
4. `WatcherRoom` 调用 `createGameOrchestrator(...).dispatch()` 或 `advanceTurn()`。
5. shared orchestration 负责：
   - 骰子与工具生成
   - 回合开始结算
   - 工具执行与地形/召唤物触发
   - 回合结束、换人、竞速完赛、结算推进
   - 事件日志与表现时间线生成
6. server 将 shared 结果回写到 Colyseus schema。
7. client 订阅 schema，同步为本地快照，再消费 shared presentation 做动画播放。

## 当前最重要的架构边界

### 1. 回合编排边界

- 只有 `packages/shared/src/gameOrchestration.ts` 可以编写回合推进逻辑。
- `WatcherRoom` 不允许再新增：
  - 下一位玩家选择
  - `roll -> action -> finishTurn -> nextTurn` 之类状态机
  - 竞速完赛后的下一回合推进逻辑
- `packages/shared/src/simulation/engine.ts` 不允许再维护第二份 turn flow。
  - 它只能包装 shared orchestration，供 golden 与本地回放使用。

### 2. 规则与表现边界

- shared 只产出语义结果：
  - 最终状态
  - 事件日志
  - `ActionPresentation`
- client 决定如何渲染这些语义事件，不在前端重写规则。

### 3. 内容与运行时边界

- 内容定义放在 `shared/content` 或 `client/game/content`。
- 行为实现放在 shared 规则层。
- `WatcherRoom` 和 `BoardScene` 都不应成为新规则的落点。

## 扩展建议

- 新增工具、地形、召唤物、角色时，先确认能否接入 shared 现有扩展点。
- 如果某个需求迫使 room 或 simulator 写规则，优先视为架构问题，而不是“先临时写进去”。
- 文档更新时，以本页作为总入口，细节再下钻到对应专题文档。
