# 角色与能力组织

本页沿用旧文件路径，内容已经切换为当前模型，不再描述旧角色系统。

## 当前模型

角色能力系统统一采用以下结构：

- `Character`
  - 角色只是能力组合体，负责声明 `skillIds`。
- `Skill`
  - 语义层概念，用来表达角色拥有哪些被动能力。
- `Modifier`
  - 逻辑层概念，是真正参与 shared 运行时的规则单元。
- `Tool`
  - 主动能力单元。阶段限定主动能力也属于 `Tool`。
- `Player.tags`
  - 所有跨回合状态的唯一归属地。

也就是：

- 设计层：`Character -> Skill`
- 运行时：`Skill -> Modifier`
- 状态层：`Player.tags`

## 当前代码入口

- 角色静态定义：`packages/shared/src/content/characters.ts`
- 角色物化与查询：`packages/shared/src/characters.ts`
- 能力运行时接口：`packages/shared/src/modifiers.ts`
- 玩家标签工具：`packages/shared/src/playerTags.ts`
- 技能模块：`packages/shared/src/skills/`
- 角色主动工具模块：`packages/shared/src/tool-modules/`

## 角色能力如何协作

### 主动能力

主动能力统一建模为 `Tool`。

- 普通行动阶段工具，例如 `bombThrow`
- 回合开始阶段工具，例如 `blazePrepareBomb`
- 会立即推进阶段的工具，例如 `volatySkipToolDie`

这些工具都通过 `useTool` 进入 shared 动作结算。

### 被动能力

被动能力统一建模为 `Skill -> Modifier`。

当前通用 hook 包括：

- `onTurnStart`
- `onTurnActionStart`
- `onTurnEnd`
- `onGetTool`
- `onDiceRoll`
- `onMovementResolved`
- `onToolUsed`
- `getMovementType`

如果未来需要新触发点，应优先扩展通用 hook，而不是在 orchestration 中加角色特判。

### 跨回合状态

所有跨回合状态都属于玩家：

- 容器：`Player.tags`
- 结构：`Tag Map`
- value 类型：`boolean | number | string`
- 命名方式：必须带技能前缀，例如 `blaze:bomb-prepared`

## 当前角色接入方式

角色文件只负责组合技能，不承载运行时状态。技能文件尽量保持高内聚，一个技能模块内集中定义：

- `SkillDefinition`
- `ModifierDefinition`
- 自己使用的 tag key
- 需要发放的工具或触发的通用效果

例如：

- `late.ts`
  - 在 `onGetTool` 时把移动改成制动。
- `blaze.ts`
  - 在 `turn-start` 发放准备炸弹工具，在 `turn-action` 阶段兑现为投弹工具。
- `volaty.ts`
  - 在 `turn-start` 发放跳过工具骰工具，在掷骰后把移动改成飞跃。

## 新增角色时的步骤

1. 在 `packages/shared/src/skills/` 新增技能模块。
2. 在模块内定义 `SkillDefinition`、`ModifierDefinition` 和 tag key。
3. 如需主动能力，新增对应 `Tool` 与执行器。
4. 在 `packages/shared/src/content/characters.ts` 中把角色和 `skillIds` 关联起来。
5. 为新机制补 golden case。

## 对 AI 的硬约束

- 不再恢复 `turnStartAction`
- 不再恢复集中式 `characterRuntime`
- 不再把跨回合状态放回角色专属状态
- 不再为单个角色在 orchestration 中添加专用字段
- 不再把复杂主动逻辑塞进 `Modifier`
