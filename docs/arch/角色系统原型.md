# 角色系统原型

## 目标

角色系统当前的重点不是做一套独立战斗框架，而是让角色能力稳定接入 shared 工具与回合系统。

## 当前入口

- 静态定义：`packages/shared/src/content/characters.ts`
- 运行时逻辑：`packages/shared/src/characterRuntime.ts`
- 回合开始动作注册：`packages/shared/src/turnStartActions.ts`

## 当前角色能力模型

角色能力分三类：

### 1. 回合开始动作

- 在 `roll` 阶段使用
- 由 `turnStartActions` 驱动
- 例子：
  - `Blaze` 准备炸弹
  - `Volaty` 跳过工具骰

### 2. 角色回合负载

- 在进入 `action` 阶段时注入额外工具
- 例子：
  - 角色专属 `Bomb Throw`
  - 角色专属强化版 `Hookshot`

### 3. 跨回合状态

- 存在 `player.characterState`
- 跟随 shared snapshot 流动
- 由 `characterRuntime.ts` 在以下时机维护：
  - 回合开始前预处理
  - 动作后更新
  - 回合结束清理
  - 非当前回合位移后的状态修正

## 当前代码边界

角色系统的真实规则不应写在：

- `WatcherRoom.ts`
- `simulation/engine.ts`
- client 组件

角色能力如果会影响真实结算，必须进入：

- `content/characters.ts`
- `characterRuntime.ts`
- 必要时再进入工具执行器或 turn-start action 注册

## 当前已落地示例

- `Blaze`
  - 准备炸弹后，下回合获得 `Bomb Throw`
- `Volaty`
  - 跳过工具骰，把当回合移动改为飞跃
- `Chain`
  - 通过跨回合状态判断是否获得小型 `Hookshot`
- `Farther`
  - 通过 `Balance` 在当前回合与下回合之间存取移动点数

## 扩展建议

如果未来角色继续增多，推荐把 `characterRuntime.ts` 逐步拆成“每个角色一个 runtime module”，但当前仍保持 shared 单点权威：

- room 只同步状态
- simulator 只调用 orchestration
- client 只展示 turn-start action 与角色说明
