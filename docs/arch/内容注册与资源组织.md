# 内容注册与资源组织

## 目标

这一层文档描述本轮重构后，项目如何把“可扩展内容”和“运行时代码”拆开，避免以后新增 Tool、角色、召唤物、地图或模型时，需要同时修改很多分散文件。

## Shared 内容注册

共享层现在把内容注册集中在 `packages/shared/src/content/`：

- `schema.ts`
  - 保存基础枚举和内容定义结构，作为注册表的共同底座
- `tools.ts`
  - 保存 Tool 注册表、Tool 骰面注册表、参数标签
- `characters.ts`
  - 保存角色注册表
- `summons.ts`
  - 保存召唤物注册表
- `effects.ts`
  - 保存表现事件类型注册表
- `defaultBoard.ts`
  - 保存默认地图布局和地形符号表

`packages/shared/src/types.ts` 不再手写一长串 union，而是从这些注册表推导：

- `ToolId`
- `CharacterId`
- `SummonId`
- `PresentationEffectType`

这样新增内容时，先改注册表，再让运行时代码读取注册表，避免“类型一份、定义一份、前后端再各写一份”的重复维护。

## Shared 规则拆分

共享规则层现在分成两类文件：

- 内容入口
  - `board.ts`
  - `tools.ts`
  - `characters.ts`
  - `summons.ts`
  - `effects.ts`
- 规则执行
  - `rules/spatial.ts`
  - `rules/actionResolution.ts`
  - `rules/actionPresentation.ts`
  - `rules/toolExecutors.ts`
  - `actions.ts`

其中：

- `actions.ts`
  - 保留统一公开入口 `resolveToolAction()`
- `toolExecutors.ts`
  - 负责各个 Tool 的直接效果
- `spatial.ts`
  - 负责平移、飞跃、推动、投射物路径等空间计算
- `actionResolution.ts`
  - 负责通用后处理，例如消耗 Tool、追加停留地形、整理返回值
- `actionPresentation.ts`
  - 负责构造 `player_motion / projectile / effect` 语义时间线

这个拆法的目标是让“新增一个 Tool”和“修改地面移动规则”尽量落在不同文件，不再挤在一个超大 `actions.ts` 中。

## Client 内容与资源清单

客户端把偏内容的配置集中到 `packages/client/src/game/content/`：

- `actionUi.ts`
  - 维护弧形 UI 的 token、颜色、交互提示、方向箭头样式映射
- `pets.ts`
  - 维护 cube-pets 模型清单、运行时 public path、原始资源来源路径

其中 `pets.ts` 同时记录：

- `publicPath`
  - 运行时由浏览器加载的路径，位于 `packages/client/public/assets/`
- `sourcePath`
  - 原始资源来源，当前指向 `resources/kenney_cube-pets_1.0/...`

这样后面替换模型或补更多角色外观时，优先改 manifest，而不是去场景组件里翻硬编码路径。

## Client 场景资产

除了 `content/` 里的纯配置，客户端现在还把“程序化几何和表现 prefab”收束到 `packages/client/src/game/assets/`：

- `board/`
  - 例如 conveyor 箭头、幸运方块、坑洞装饰、tile 组合体
- `previews/`
  - 例如落点圈、爆炸预览块、建墙 ghost
- `summons/`
  - 例如钱包，以及 summon id 到组件的 registry
- `presentation/`
  - 例如篮球投射物、火箭投射物、火箭爆炸，以及语义事件到组件的 registry
- `player/`
  - 例如玩家脚下光环和当前回合标记

这层的意图是把 `BoardScene.tsx` 从“既管交互，又管每个 mesh 长什么样”改成“管理何时显示哪些资产”。  
新增视觉资产时，优先新增独立 asset 文件，再接到对应 registry，而不是继续往 `BoardScene.tsx` 塞 JSX 几何。

## Server 房间拆分

服务端房间层现在把“房间编排”和“状态映射/落库”拆开：

- `WatcherRoom.ts`
  - 负责消息入口、回合推进、调用共享结算
- `roomStateMappers.ts`
  - 把 Colyseus schema 状态转换成共享层快照
- `roomStateMutations.ts`
  - 把共享层结算结果写回 schema
- `roomEventLog.ts`
  - 把共享层触发结果转换成房间事件日志

这让 `WatcherRoom` 更接近 orchestration，而不是同时承担 mapper、repository、event formatter 三种职责。

## 当前资源约定

当前建议的资源组织约定如下：

- 原始第三方或待处理素材
  - 放在仓库根目录 `resources/`
- 浏览器直接访问的运行时资源
  - 放在 `packages/client/public/assets/`
- 用于把资源接入游戏的 manifest
  - 放在 `packages/client/src/game/content/`
- 用于驱动规则的玩法内容
  - 放在 `packages/shared/src/content/`

## 新增内容时的入口

新增 Tool：

1. 在 `packages/shared/src/content/tools.ts` 注册定义
2. 如需出现在工具骰中，在同文件补骰面
3. 在 `packages/shared/src/rules/toolExecutors.ts` 补执行器
4. 如需前端专属展示，再在 `packages/client/src/game/content/actionUi.ts` 补 UI 元数据

新增角色：

1. 在 `packages/shared/src/content/characters.ts` 注册
2. 如有新主动技能 Tool，仍按 Tool 流程接入

新增召唤物：

1. 在 `packages/shared/src/content/summons.ts` 注册
2. 在共享规则里补触发逻辑
3. 在 `BoardScene.tsx` 增加渲染映射

新增地形或地图：

1. 在 `packages/shared/src/content/defaultBoard.ts` 或未来其他地图内容文件中补布局
2. 在 `terrain.ts` 补地形行为
3. 在 `BoardScene.tsx` 补场景表现

新增模型资源：

1. 把原始素材放进 `resources/`
2. 把运行时资源放进 `packages/client/public/assets/`
3. 在 `packages/client/src/game/content/` 的 manifest 中登记

## 现阶段边界

当前仍然保留了少量“内容 -> 表现”的显式映射，例如：

- `BoardScene.tsx` 内的地形与召唤物渲染
- `SceneDirectionArrows.tsx` 内的方向箭头样式

这部分还没有完全插件化，但注册表已经把最容易分散的 ID、参数、骰面和资源路径先收束起来了。后续如果内容量继续上升，可以再把场景表现映射继续拆到更细的 registry 中。
## 2026-03-31 Client Composition Follow-up

Beyond content manifests and scene assets, the client now also separates "composition" modules:

- `components/HudSidebar.tsx` for sidebar composition
- `hooks/useKeyboardInteraction.ts` for keyboard shortcuts
- `hooks/useAnimationClock.ts` for local animation time
- `hooks/useAutomationBridge.ts` for `render_game_to_text()` and `advanceTime()`
- `state/roomCommands.ts` for local command guards before sending room actions

These are not content registries themselves, but they keep high-churn UI and orchestration code out of large monolithic entry files.
