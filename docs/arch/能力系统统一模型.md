# 能力系统统一模型

## 目的

本文档定义 shared 层能力系统的唯一权威模型。未来的 `Tool`、`Interaction`、`Skill`、`Modifier`、`Character`、`Player.tags`、`Turn Phase` 都必须以本页为准。

目标只有四个：

- 高内聚
- 低耦合
- 强可扩展性
- 开闭原则

只要某个实现方案偏离这四个目标，就不应直接落地。

## 权威性

未来 AI 在能力系统相关开发中，未经明确批准，不应：

- 引入与 `Tool / Interaction / Skill / Modifier / Character / Player Tags / Turn Phase` 平行的新顶层概念
- 恢复 `turnStartAction`、集中式 `characterRuntime`、角色专属运行时状态等旧模型
- 在 room、simulator 或 client 中重写 shared 规则
- 为单个工具、单个技能、单个角色向主流程里塞专用变量

如果出现本文档无法表达的新需求，正确做法是先更新本文档，再改代码。

## 顶层概念

### Player

`Player` 是真实对局状态的拥有者。

- 玩家拥有位置、工具、阶段信息、标签等通用状态。
- 所有跨回合状态都属于玩家，而不是角色。
- 角色只为玩家提供能力配置，不拥有独立运行时状态容器。

### Tool

`Tool` 是玩家主动操控的道具或主动能力单元。

每个 `Tool` 必须同时定义两部分：

- `interaction`
  - 描述玩家如何提供输入。
- `execute`
  - 描述它如何影响规则层。

硬约束：

- 任何位移都必须通过 `movementSystem` 施加。
- `Tool` 不能直接修改玩家坐标。
- 复杂主动逻辑优先放进 `Tool`，不要塞进 `Modifier`。

### Interaction

`Interaction` 是 `Tool` 的输入契约。

- 它只描述“玩家如何操作这个工具”。
- 它不是规则概念，不承载游戏效果。
- 规则结算仍由 `Tool.execute` 决定。

### Skill

`Skill` 是语义层概念，是角色能力在设计层和内容层的名字。

- `Skill` 用来表达“这个角色有哪些被动能力”。
- `Skill` 本身不直接承担复杂运行时逻辑。
- `Skill` 的运行时实现由 `Modifier` 承担。

### Modifier

`Modifier` 是能力系统的逻辑层基元，是 shared 规则真正执行的单位。

- 一个 `Skill` 可以映射到一个或多个 `Modifier`。
- `Modifier` 通过通用 hook 参与 shared 流程。
- `Modifier` 可以读取和写入 `Player.tags`。

当前确认需要覆盖的通用 hook 包括：

- `onTurnStart`
- `onTurnActionStart`
- `onTurnEnd`
- `onGetTool`
- `onDiceRoll`
- `onMovementResolved`
- `onToolUsed`
- `getMovementType`

原则：

- hook 必须是通用的，不能为单个技能临时发明专用类型。
- hook 既可以做事件响应，也可以做通用变换。

### Character

`Character` 是 `Skill` 的集合体。

- 角色只负责组合能力。
- 角色定义尽量保持声明式。
- 角色不再拥有单独的运行时状态容器。

### Player Tags

玩家公共状态容器统一为 `Tag Map`。

- 结构：`Player.tags: Partial<Record<string, PlayerTagValue>>`
- value 类型：`boolean | number | string`
- 命名规则：必须带技能前缀，例如 `blaze:bomb-prepared`

硬约束：

- 不再新增角色专属状态字段。
- 不再新增工具专属状态字段。
- 不再新增“某角色状态 map”之类的并行容器。

如果未来 `Tag Map` 无法表达某种结构化状态，必须先更新本文档，再新增能力模型。

### Turn Phase

回合阶段统一为三个：

- `turn-start`
- `turn-action`
- `turn-end`

默认边界：

- `rollDice`：从 `turn-start` 进入 `turn-action`
- `endTurn`：从 `turn-action` 进入 `turn-end`
- `advanceTurn`：在 `turn-end` 结算完成后进入下一名玩家的 `turn-start`

补充约束：

- 每个阶段都允许使用 `Tool`
- `turn-end` 默认没有工具，但模型本身允许存在阶段限定工具
- Blaze、Volaty 这类回合开始能力必须建模为 `turn-start` 阶段工具
- 阶段工具如需在使用后立刻推进阶段，必须通过通用 phase effect 表达，而不是重新发明概念

## 规则边界

### shared orchestration

回合推进、阶段切换、统一命令入口都只能写在：

- `packages/shared/src/gameOrchestration.ts`

未来 AI 不应在以下位置编写回合逻辑：

- `packages/server/src/rooms/WatcherRoom.ts`
- `packages/shared/src/simulation/engine.ts`

### movementSystem

所有真实位移都必须走：

- `packages/shared/src/rules/movementSystem.ts`

任何 `Tool` 或 `Modifier` 都不应直接修改玩家位置。

### room / simulator / client

- room 只做 authoritative adapter
- simulator 只做本地包装与测试驱动
- client 只做表现、输入、预览与播放

它们都不应成为第二套能力规则实现。

## 资源组织原则

能力系统按“完整能力单元”组织，而不是按零散表格与分支组织。

推荐方向：

- 一个 Tool 尽量在一个模块内收拢：
  - `definition`
  - `interaction`
  - `execute`
- 一个 Skill 对应的 Modifier 定义尽量收敛在一个模块内：
  - `skill metadata`
  - `modifier hooks`
  - `tag key`
- Character 文件只做组合，不写运行时细节

## 当前实现约束

当前代码已经完成以下收口：

- `turnStartAction` 已吞并进 `useTool`
- 集中式 `characterRuntime` 已删除
- 跨回合状态已统一迁移到 `Player.tags`
- 角色能力统一改为 `Character -> Skill -> Modifier`
- `Blaze`、`Volaty` 的回合开始能力已改为 `turn-start` 工具

未来不应恢复旧入口，也不应保留双轨系统。

## 对未来 AI 的明确禁止

未经批准，不应：

- 新建 `Ability`、`PassiveAction`、`TurnStartAction2`、`CharacterRuntimeModule` 之类与本文档平行的新概念
- 为单个角色在 orchestration 中加入专用布尔变量
- 为单个技能增加主流程特判字段
- 把复杂主动逻辑写进 `Modifier` 而不是 `Tool`
- 把自动触发的被动逻辑写进 `Tool` 而不是 `Modifier`
- 在 room 或 simulator 中补一份 shared 规则

## 已知风险

- `Tag Map` 适合布尔、数值和短字符串状态，不适合无约束的复杂嵌套对象。
- `Modifier` hook 必须保持通用。如果为了某个技能临时发明专用 hook，系统会退化回旧的分支中心。
- 如果 `Tool` 和 `Modifier` 分工不清，主动逻辑和被动逻辑会重新缠在一起。

## 决策检查表

每次修改能力系统前，先检查：

1. 这是主动操作，还是自动触发。
2. 如果是主动操作，它是不是 `Tool`。
3. 如果是自动触发，它是不是 `Modifier` hook。
4. 它的状态是不是属于 `Player.tags`。
5. 它有没有绕开 `movementSystem`。
6. 它有没有偷偷引入本文档之外的新概念。

只要第 6 条的答案是“有”，就不应继续实现，必须先更新本文档。
