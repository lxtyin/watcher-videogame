# 能力系统统一模型

## 目的

本文档是当前能力系统的唯一权威定义。它同时覆盖原来的“能力系统统一模型”和“角色系统原型”，未来 AI 修改角色、技能、工具或 modifier 时，只应参考本文，不应再恢复第二份并行文档。

能力系统的目标只有四个：

- 高内聚
- 低耦合
- 强可扩展性
- 开闭原则

## 权威边界

未经批准，未来 AI 不应：

- 引入与 `Tool / Interaction / Skill / Modifier / Character / Player.tags / Player.modifiers / Turn Phase` 平行的新顶层概念
- 恢复 `turnStartAction`、集中式 `characterRuntime`、角色专属运行时状态容器
- 在 room、simulator 或 client 中补一份 shared 能力规则
- 为单个工具、单个技能或单个角色向主流程塞专用变量

如果出现本文档无法表达的新需求，正确做法是先更新本文档，再改代码。

## 顶层概念

### Player

`Player` 是真实对局状态的拥有者。

- 玩家拥有位置、工具、阶段信息、标签与运行时 modifier
- 所有跨回合状态都属于玩家，而不是角色
- 角色只提供能力配置，不拥有独立运行时状态容器

### Character

`Character` 是 `Skill` 的集合体。

- 角色定义保持声明式
- 角色文件只负责组合能力
- 角色不再承载运行时状态

### Skill

`Skill` 是设计层概念，用来表达角色拥有哪些被动能力。

- `Skill` 用于内容组织与语义命名
- `Skill` 本身不承担复杂运行时逻辑
- `Skill` 的运行时实现由一个或多个 `Modifier` 承担

### Modifier

`Modifier` 是能力系统的逻辑层基元，是 shared 规则真正执行的单位。

- `Modifier` 通过通用 hook 参与 shared 流程
- `Modifier` 可以读写 `Player.tags`
- 运行时可插拔 `Modifier` 必须显式安装到 `Player.modifiers`

当前通用 hook：

- `onTurnStart`
- `onTurnActionStart`
- `onTurnEnd`
- `onGetTool`
- `onDiceRoll`
- `onMovementResolved`
- `onToolUsed`
- `getMovementType`

约束：

- hook 必须是通用的，不能为单个技能临时发明专用类型
- 复杂主动逻辑不应写进 `Modifier`

### Tool

`Tool` 是玩家主动使用的能力单元。

每个 `Tool` 必须同时定义：

- `definition`
- `interaction`
- `isAvailable`
- `execute`
- `getTextDescription`

约束：

- 任何位移都必须通过 `movementSystem`
- `Tool` 不能直接修改玩家坐标
- 复杂主动逻辑优先放进 `Tool`
- `Tool` 不再使用统一的 `ToolCondition` 声明系统；工具自己的可用性由 `ToolContentDefinition.isAvailable` 裁决
- 工具参数只保留少量通用字段，例如 `movePoints`；工具私有参数由工具模块内部约定命名
- `Tool` 的前端展示文本必须由自己的 `getTextDescription` 手写，不应依赖共享参数标签表自动生成

### Interaction

`Interaction` 是 `Tool` 的输入契约。

- 它只描述玩家如何提供输入
- 它不描述规则效果
- 合法范围、preview 内容与最终效果由工具自己的规则逻辑决定

### Player.tags

玩家公共状态容器统一为 `Tag Map`。

- 结构：`Player.tags: Partial<Record<string, PlayerTagValue>>`
- value 类型：`boolean | number | string`
- 命名规则：必须带技能或机制前缀，例如 `blaze:bomb-prepared`

约束：

- 不再新增角色专属状态字段
- 不再新增工具专属状态字段
- 不再新增并行状态容器
- 不再把“Modifier 是否存在”编码进 tags

### Player.modifiers

运行时插拔的被动效果统一存放在 `Player.modifiers`。

- 结构：`Player.modifiers: ModifierId[]`
- 含义：表示“这个玩家当前真正装着哪些运行时 Modifier”
- 职责：负责 modifier 的安装、移除与生命周期管理

约束：

- 不允许通过枚举全部 modifier 再结合 tags 推断谁生效
- 角色静态自带的 modifier 仍由 `Character -> Skill -> Modifier` 推导

### Turn Phase

回合阶段统一为三个：

- `turn-start`
- `turn-action`
- `turn-end`

默认边界：

- `rollDice`：从 `turn-start` 进入 `turn-action`
- `endTurn`：从 `turn-action` 进入 `turn-end`
- `advanceTurn`：在 `turn-end` 结算后进入下一名玩家的 `turn-start`

补充约束：

- 每个阶段都允许使用 `Tool`
- 回合开始能力如果需要玩家主动确认，应建模成 `turn-start` 工具

## 角色如何组织

当前角色组织方式统一为：

- 设计层：`Character -> Skill`
- 运行时：`Skill -> Modifier`
- 主动能力：`Tool`
- 状态层：`Player.tags + Player.modifiers`

当前入口文件：

- 角色静态定义：`packages/shared/src/content/characters.ts`
- 角色物化与查询：`packages/shared/src/characters.ts`
- 技能模块：`packages/shared/src/skills/`
- modifier 接口：`packages/shared/src/modifiers.ts`
- 玩家标签工具：`packages/shared/src/playerTags.ts`
- 角色主动工具：`packages/shared/src/tool-modules/`

组织原则：

- 一个技能模块尽量在一个文件内收拢：
  - `SkillDefinition`
  - `getTextDescription`
  - `ModifierDefinition`
  - 自己使用的 tag key
  - 需要安装或移除的 runtime modifier
  - 需要发放的工具或触发的通用效果
- 角色文件只做组合，不写运行时细节

## 新增角色或技能时的推荐步骤

1. 在 `packages/shared/src/skills/` 新增或复用技能模块。
2. 在模块内定义 `SkillDefinition`、`getTextDescription`、`ModifierDefinition` 和相关 tag key。
3. 如需主动能力，再补对应 `Tool` 模块。
4. 在 `packages/shared/src/content/characters.ts` 里把角色与 `skillIds` 关联起来。
5. 为新机制补 golden case。

## 规则边界

### shared orchestration

回合推进、阶段切换与统一命令入口只能写在：

- `packages/shared/src/gameOrchestration.ts`

不得在以下位置编写回合逻辑：

- `packages/server/src/rooms/WatcherRoom.ts`
- `packages/shared/src/simulation/engine.ts`

### movementSystem

所有真实位移都必须走：

- `packages/shared/src/rules/movementSystem.ts`

任何 `Tool` 或 `Modifier` 都不应直接修改玩家位置。

### room / simulator / client

- room 只做 authoritative adapter
- simulator 只做本地包装与测试驱动
- client 只做输入、预览、播放与渲染

它们都不应成为第二套能力规则实现。

## 对未来 AI 的明确禁止

未经批准，不应：

- 新建 `Ability`、`PassiveAction`、`TurnStartAction2`、`CharacterRuntimeModule` 之类的新并行概念
- 为单个角色在 orchestration 中加入专用布尔变量
- 为单个技能增加主流程特判字段
- 把复杂主动逻辑写进 `Modifier`
- 把自动触发的被动逻辑写进 `Tool`
- 在 room 或 simulator 中补一份 shared 规则

## 决策检查表

每次修改能力系统前，先检查：

1. 这是主动操作，还是自动触发。
2. 如果是主动操作，它是不是 `Tool`。
3. 如果是自动触发，它是不是 `Modifier` hook。
4. 它的状态是不是属于 `Player.tags`，或者它的运行时装配是不是属于 `Player.modifiers`。
5. 它有没有绕开 `movementSystem`。
6. 它有没有偷偷引入本文档之外的新概念。

只要第 6 条的答案是“有”，就不应继续实现，必须先更新本文档。
