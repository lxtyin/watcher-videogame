# 房间与大厅流程

## 当前目标

这一轮把原型从“进入页面立即联机并开局”扩展为完整的房间流：

- 主页
- 创建房间
- 加入房间
- 大厅准备
- 房主开始游戏
- 游戏结算后回到房间

这一步仍然是原型实现，重点是把状态边界理顺，而不是做账号系统或公网匹配。

## 页面入口

当前客户端有两个主要入口：

- `/`
  - 主页
  - 提供“创建房间 / 加入房间”两个入口
- `/?room=<roomCode>`
  - 房间页
  - 负责大厅、对局和结算后的回房

当前 `roomCode` 直接复用 Colyseus 的 `roomId`。
对用户来说它就是房间号，对实现来说不需要额外维护一张 roomCode 到 roomId 的映射表。

## 房间状态模型

共享快照和服务端 schema 现在都显式表达房间层状态：

- `roomCode`
- `roomPhase`
  - `lobby`
  - `in_game`
  - `settlement`
- `hostPlayerId`
- `players[].isReady`
- `players[].isConnected`

这层状态和原有的玩法状态并列存在：

- 房间层决定能不能加入、能不能开始、能不能返回大厅
- 游戏层决定当前回合、工具、地形、结算

这样可以避免把 lobby 逻辑塞进 `turnInfo` 里。

## 服务端房间生命周期

`WatcherRoom` 当前按下面的阶段流转：

1. `onCreate`
   - 根据地图建立棋盘
   - 初始化 `roomCode / roomPhase / hostPlayerId`
   - 初始阶段固定为 `lobby`
2. `onJoin`
   - 新玩家加入时进入大厅
   - 首个玩家自动成为房主
   - 不再“加入即开局”
   - 如果是重连，则认领原有 seat，而不是创建新玩家
3. `startGame`
   - 仅房主可触发
   - 仅 `lobby` 阶段可触发
   - 要求所有在线玩家都已准备
   - 开始后房间 `lock()`
4. `in_game`
   - 继续沿用现有回合规则
   - 只有这一阶段允许玩家行动
5. `settlement`
   - 竞速模式全部完赛后进入
   - 保留棋盘和结算信息，等待玩家操作
6. `returnToRoom`
   - 从 `settlement` 回到 `lobby`
   - 重置对局态
   - 房间 `unlock()`

## Lobby 规则

大厅当前支持：

- 显示房间号、地图、模式、房主
- 显示所有玩家的棋子缩略图、用户名、角色、在线状态、准备状态
- 玩家可在大厅切换角色
- 玩家可在自己的卡片上点击准备/取消准备
- 房主在所有在线玩家准备后点击开始游戏

当前默认认为“在线玩家”才参与 ready 检查。
断线玩家会先进入短暂重连保留窗口，超时后再被移除。

## 对局开始与回房

开始游戏时会做一次完整的 match reset：

- 重置棋盘到地图初始状态
- 清空召唤物
- 清空临时工具、turn flags、表现缓存
- 按地图出生点重新摆放玩家
- 保留玩家的用户名、颜色、角色选择
- 清空 `finishRank / finishedTurnNumber`

回到房间时也会走同样的 reset，但目标阶段是 `lobby`。

## 刷新恢复策略

当前不做账号系统，但实现了一个“相对持久”的本地恢复方案：

### 浏览器本地持久化

客户端会把下面两类信息写到 `localStorage`：

- 上次输入的用户名
- 最近一次房间会话
  - `roomCode`
  - `reconnectionToken`
  - `playerName`

### 服务端重连保留

服务端在非主动离房时调用 `allowReconnection()`：

- 玩家刷新页面或短暂断线时，不会立刻丢失 seat
- 浏览器重新加载后，客户端会优先尝试 `client.reconnect(reconnectionToken)`
- 成功后继续认领原玩家 id、原角色、原颜色和原对局状态

### 当前边界

这个方案解决的是“同一浏览器刷新或短线恢复”。
它不解决：

- 跨设备恢复
- 清空浏览器存储后的恢复
- 很长时间离线后的恢复

如果后续需要更强的持久化，再考虑引入账号、游客 token 或服务端持久 seat。

## 客户端结构

客户端目前按下面的职责拆分：

- `App.tsx`
  - 负责主页/房间页路由切换
  - 负责 create/join/reconnect/leave 命令接线
- `useWatcherConnection.ts`
  - 负责 Colyseus 连接
  - 负责 `localStorage` 会话恢复
  - 负责把 schema 同步到 store
- `HudSidebar.tsx`
  - 根据 `roomPhase` 在 lobby 与 in-game HUD 之间切换
- `RaceSettlementOverlay.tsx`
  - 在 `settlement` 阶段展示竞速结算

## 当前限制

- 房间号目前直接使用 Colyseus `roomId`
- 还没有主页上的“房间列表”或“最近房间”
- 还没有踢人、换房主、观战位
- 还没有专门的断线提示倒计时 UI

这些都可以继续建立在现在的 `roomPhase + host + ready + reconnect` 这套模型上扩展。
