# 房间与大厅流程

## 房间阶段

`roomPhase` 当前有三个阶段：

- `lobby`
- `in_game`
- `settlement`

## lobby

`lobby` 由 `packages/server/src/rooms/WatcherRoom.ts` 负责，当前包含：

- 玩家加入房间
- 设置昵称与宠物
- 选择角色
- 切换 ready
- 房主开始游戏
- 房主踢人
- 断线重连

这一阶段不进入 shared turn orchestration。

client 在 `roomPhase === "lobby"` 时渲染独立的 `LobbyScreen`，不挂载对局地图与对局 HUD。大厅页面按玩家卡片展示房间成员，卡片内包含棋子、玩家配色、名称、在线状态、准备状态、准备按钮与角色卡。点击自己的角色卡会打开选角弹窗，弹窗按四列网格展示全部角色卡并通过 `setCharacter` 切换自己的角色。新加入玩家的默认角色为鹅哈哈。

## 开局

开局时 room 会：

1. 重置对局 runtime 与棋盘可变状态。
2. 按地图出生点重置玩家位置。
3. 清空事件日志与表现状态。
4. 把 `roomPhase` 设为 `in_game`。
5. 调用 shared `advanceTurn()` 开启首回合。

从这里开始，回合推进完全由 `packages/shared/src/gameOrchestration.ts` 接管。

## 对局阶段

client 只发送高层命令：

- `rollDice`
- `useTool`
- `endTurn`
- `setCharacter`
- `grantDebugTool`

`WatcherRoom` 只做：

- command 收包
- schema -> shared 映射
- 调用 `dispatch(command)`
- shared -> schema 回写

其中 `endTurn` 当前分为两段：

- 当当前阶段是 `turn-action` 时，`endTurn` 先把玩家推进到 `turn-end`
- 如果 `turn-end` 阶段发放了可用工具，则 client 继续通过 `useTool` 使用它们，或再次发送 `endTurn` 跳过该阶段
- 只有当 `turn-end` 真正结束后，shared 才会切到下一名玩家

如果 shared 返回待推进状态，room 只按 `latestPresentation.durationMs` 挂 timer，随后调用 `advanceTurn()`。

- `pendingAdvance` 不只用于 race 终点动画，也用于 bedwars 的最终淘汰结算
- 当 shared 需要“先播完表现，再切阶段”时，会继续保持 `roomPhase = in_game`
- room 侧仍然只按 `latestPresentation.durationMs` 设 timer，时间到后统一调用 `advanceTurn()`
- 真正切入 `settlement` 的时机仍由 shared 决定，server 不额外推断“是否该现在结算”

## 结算阶段

当前结算主要用于 race 模式：

- 所有玩家完赛后，shared 将快照推进到 `settlement`
- client 根据 `settlementState === "complete"` 显示结算覆盖层
- 玩家可通过 `returnToRoom` 回到原房间大厅

## 回房

`returnToRoom` 会：

- 把 `roomPhase` 设回 `lobby`
- 重置 match runtime
- 保留房间与玩家座位
- 清空 ready 状态

## 对未来 AI 的约束

- 大厅与房间生命周期改动，可以在 `WatcherRoom.ts` 内处理
- 一旦涉及回合推进、阶段切换、换人、终点推进、停留结算，必须回到 `packages/shared/src/gameOrchestration.ts`
- 不要在房间层写临时 turn 特判
