# 房间与大厅流程

## 当前目标

这一层负责把原型从“打开页面直接进房间”整理成完整的多人流程：

- 主页
- 创建房间
- 加入房间
- 大厅准备
- 房主开始游戏
- 游戏结算后回到同一个房间

当前仍然是原型实现，重点是让状态边界、页面入口和本地持久化足够清晰，方便继续扩展。

## 页面入口

当前客户端有三种可见入口状态：

- `/`
  - 主页
  - 居中展示大标题
  - 下方是一张玩家资料卡，用于设置用户名和棋子
  - 再下方是两个大卡片入口：`创建房间`、`加入房间`
- `/?screen=create`
  - 创建房间页
  - 仍然保持居中布局
  - 页面中间展示地图缩略图
  - 地图名在上方，模式和简介在下方
  - 支持左右切换和滑动切换地图
- `/?room=<roomCode>`
  - 房间页
  - 负责大厅、对局和结算后的回房

## 主页交互

主页现在把“玩家资料”和“房间动作”拆开：

- 玩家资料卡
  - `playerName`
  - `petId`
- 创建房间
  - 点击后进入独立地图选择页
- 加入房间
  - 不再跳转新页面
  - 点击后在主页原地展开房间号输入框
  - 提交时直接复用上方资料卡中的用户名和棋子

用户名和棋子在没有历史记录时都会生成随机初始值，并立即写入本地存储。

## 地图选择页

创建房间页只做一件事：选择地图并创建房间。

- 地图数据直接读取 shared 的地图注册表
- 缩略图由地图布局动态生成，而不是维护一套单独图片
- 这样地图的玩法配置、缩略图和实际运行时棋盘保持同源

当前地图展示信息包括：

- `label`
- `mode`
- `description`

## 房间状态模型

共享快照和服务端 schema 当前都显式维护房间层状态：

- `roomCode`
- `roomPhase`
  - `lobby`
  - `in_game`
  - `settlement`
- `hostPlayerId`
- `players[].isReady`
- `players[].isConnected`
- `players[].petId`

游戏玩法状态仍然和房间状态并列存在：

- 房间层决定是否允许加入、准备、开始、回房
- 游戏层决定回合、工具、地形、结算

## Lobby 规则

大厅当前支持：

- 显示房间号、地图、模式、房主
- 显示玩家缩略图、用户名、角色、在线状态、准备状态
- 玩家在自己的卡片上切换准备状态
- 玩家在大厅阶段切换角色
- 所有在线玩家准备后，只有房主可以开始游戏

## 对局与回房

开始游戏时会执行一次完整的 match reset：

- 重置棋盘到地图初始状态
- 清空召唤物、临时工具、turn flags、表现缓冲
- 按地图出生点重新摆放玩家
- 保留用户名、棋子、颜色和角色选择

竞速模式全部完赛后进入 `settlement`。  
玩家点击“回到房间”后，房间重新回到 `lobby` 并重新开放加入。

## 本地持久化策略

当前不做账号系统，但做了两层本地持久化：

### 玩家资料

用于主页和创建房间页复用：

- `playerName`
- `petId`

即使当前不在房间里，刷新后也能保留这两项资料。

### 房间会话

用于刷新后认领原来的 seat：

- `roomCode`
- `reconnectionToken`
- `playerName`
- `petId`

服务端在非主动离房时调用 `allowReconnection()`，客户端重新进入房间路由时优先尝试 `client.reconnect()`。

## 客户端职责拆分

- `App.tsx`
  - 管理主页、创建页、房间页路由状态
  - 管理当前本地玩家资料
- `useWatcherConnection.ts`
  - 负责 create / join / reconnect / leave
  - 负责本地资料和房间会话持久化
- `HomeScreen.tsx`
  - 负责主页资料卡和两张入口卡片
- `CreateRoomScreen.tsx`
  - 负责地图选择和房间创建
- `HudSidebar.tsx`
  - 根据 `roomPhase` 切换大厅和对局 HUD

## 当前限制

- 房间号仍直接使用 Colyseus `roomId`
- 还没有房间列表、最近房间或公开匹配
- 还没有跨设备身份恢复
- 还没有断线倒计时和更完整的大厅管理功能

这些能力都可以继续建立在当前的 `roomPhase + host + ready + reconnect + playerProfile` 模型上扩展。
## 2026-04-04 Lobby Polish

- `房间玩家` 列表现在支持房主直接踢人。
- 踢人通过独立的 `kickPlayer` 房间消息完成，服务端立即移除目标 seat。
- 被踢客户端会清掉本地房间会话，并回到房间入口页显示原因提示。
- 这条链路和普通离房分开建模，便于后续继续扩展禁赛、房主转移等房间控制能力。
## 2026-04-04 Mobile Room Layout

- `app-shell` for in-room play now keeps a left sidebar plus scene layout even under mobile breakpoints.
- The sidebar can be collapsed from the shared room shell on both desktop and mobile.
- Mobile portrait now shows a rotate-device overlay instead of trying to squeeze the gameplay layout vertically.
- Camera input mapping now treats:
  - single touch as gameplay interaction only
  - two-finger touch as camera rotate + zoom
