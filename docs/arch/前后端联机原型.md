# 前后端联机原型

## 服务端

### 位置

- `packages/server/src/index.ts`
- `packages/server/src/schema/WatcherState.ts`
- `packages/server/src/rooms/WatcherRoom.ts`

### 当前职责

- 创建 `watcher_room`
- 初始化权威棋盘状态
- 处理玩家加入、离开与回合切换
- 校验 `rollDice`、`useTool`、`endTurn`
- 为当前行动玩家生成本回合 `tools[]`
- 将共享层 Tool 结算结果写回房间状态
- 维护事件日志和最近回合信息

### 状态同步模型

服务端状态根节点是 `WatcherState`，包含：

- `board`
  - Colyseus `MapSchema`，保存地块状态。
- `players`
  - Colyseus `MapSchema`，保存玩家位置和当前回合 Tool 列表。
- `turnInfo`
  - 保存当前玩家、回合阶段、最近一次掷骰结果和回合提示信息。
- `eventLog`
  - 保存最近几条行为日志。

当前 `players[playerId].tools` 中的每一项都是一个 `TurnToolState`：

- `instanceId`
  - 当前回合内唯一的 Tool 实例 id。
- `toolId`
  - Tool 类型，例如 `movement`、`jump`、`dash`。
- `charges`
  - 预留给多次使用 Tool 的统一字段。
- `paramsJson`
  - 当前 Tool 参数包的 JSON 字符串，客户端和共享层会解析为统一的 `params`。
- `spawnX / spawnY`
  - 玩家出生点，供坑洞等停留地形直接复位。
- `turnFlags`
  - 当前回合的地形触发标记，例如幸运方块的限次标记。

### 当前服务端流程

1. 玩家加入后分配出生点。
2. 当前玩家进入 `roll` 阶段。
3. 收到 `rollDice` 后，服务端生成：
   - 一个 `Movement(points)` Tool
   - 一个来自可配置工具骰面的基础 Tool
4. 服务端把这两个 Tool 实例写入当前玩家的 `tools[]`，并切换到 `action` 阶段。
5. 收到 `useTool` 后，服务端按 `toolInstanceId` 找到当前激活的 Tool。
6. 服务端根据 Tool 的目标模式读取 `direction` 或 `targetPosition`。
7. 服务端调用共享层 `resolveToolAction()` 完成权威结算。
8. 服务端将新的位置、受影响玩家、地块变化、地形触发结果和新的 `tools[]` 写回房间状态。
9. 收到 `endTurn` 后，服务端清空本回合 Tool 列表，切换到下一名玩家并回到 `roll` 阶段。
10. 若收到 `grantDebugTool`，服务端会在当前行动玩家的回合列表里追加指定 Tool。

## 客户端

### 位置

- `packages/client/src/App.tsx`
- `packages/client/src/game/network/useWatcherConnection.ts`
- `packages/client/src/game/state/useGameStore.ts`
- `packages/client/src/game/components/BoardScene.tsx`
- `packages/client/src/game/components/SceneInteractionHud.tsx`
- `packages/client/src/game/utils/boardMath.ts`

### 当前职责

- 与 Colyseus 房间建立连接
- 将房间状态反序列化为前端快照
- 保存 HUD 和 3D 交互所需的本地状态
- 基于当前 `tools[]` 自动选择默认可用 Tool
- 发送掷骰、使用 Tool 和结束回合请求
- 在 3D 场景中提供按下 Tool、拖动选目标、松手执行、右键取消的交互
- 用共享层规则生成客户端动作预览

### 当前交互模型

`App.tsx` 负责 HUD 和全局输入：

- 展示中文化后的本回合掷骰结果和当前 `tools[]`
- 维护当前选中的 `toolInstanceId`
- 将不可用 Tool 置灰，并在点击时弹出提示信息
- 在左侧栏提供调试下拉列表，可直接发放任意已实现 Tool
- 提供掷骰、结束回合和键盘兜底输入

`BoardScene.tsx` 负责 3D 棋盘交互：

- 在当前玩家棋子头顶渲染弧形 Tool 环
- 动态渲染当前回合可见的 `tools[]`
- `Movement` Tool 在按钮上直接显示点数
- 条件不满足的 Tool 以灰态展示
- 对定向 Tool 显示场景内 3D 四向箭头
- 对选格 Tool 按目标采样模式吸附和高亮目标格
  - `axis_line` 用于 `Brake`
  - `adjacent_ring` 用于 `BuildWall`
  - `board_any` 用于 `Teleport`
- 为坑洞、幸运方块和加速带提供程序化 3D 地形表现
- 加速带直接在棋盘上显示方向箭头，而不是只放在 HUD 中
- 用 `cube-pets` GLB 模型替换早期椭球棋子，并按同格人数做竖直叠放
- 用最近一次位移方向驱动角色朝向，让移动后的模型朝向与操作结果一致
- 拖动自己的棋子来选择方向或格子
- 鼠标中键按住时旋转视角，滚轮控制缩放
- 进入瞄准时隐藏弧形 Tool 环，执行或取消后再显示
- 左键松开执行当前 Tool，右键取消本次瞄准
- 在预计落点或预计命中玩家脚底下显示预览圈
- 使用共享层结算结果生成路径、高亮区域和建造虚影预览
- `Rocket` 预览会把预计爆炸范围渲染成更醒目的红色地块标记

### 当前客户端状态流

1. `useWatcherConnection.ts` 监听 Colyseus 房间状态。
2. `deserializeRoomState.ts` 将 Schema 转成前端快照。
3. 客户端根据当前行动玩家的 `tools[]` 自动挑选默认可用 Tool。
4. HUD 与场景根据 `selectedToolInstanceId` 渲染当前交互模式。
5. 客户端执行动作前，会使用共享层 `resolveToolAction()` 生成预览。
6. 客户端把 `toolInstanceId` 和可选方向或目标格发给服务端，由服务端做最终裁定。
7. 新状态同步回来后，客户端刷新 Tool 环、3D 箭头、日志和侧边栏说明。

## 自动化桥接

客户端当前暴露了这些测试接口：

- `window.render_game_to_text`
  - 输出当前可玩状态的 JSON 文本，供自动化读取。
- `window.advanceTime`
  - 允许测试脚本稳定推进时间。
- `window.project_grid_to_client`
  - 把棋盘坐标投影到屏幕坐标，方便自动化脚本稳定点击或拖动场景对象。

这些桥接接口用于 Playwright 烟测与交互回归验证。

## 当前已验证行为

- 客户端可以连接房间并进入 `roll -> action -> endTurn` 流程
- 当前玩家掷骰后会得到一份统一的 `tools[]`
- `Movement` 已经和其他 Tool 一样通过 `toolInstanceId` 被选择和消耗
- `Dash` 会给当前所有 `Movement` +2，且在没有 `Movement` 时会禁用
- `Brake` 会根据拖拽目标格高亮并执行实际可达的落点
- `BuildWall` 会在周围八格高亮并预览土墙虚影
- `Basketball` 会沿方向飞行、遇墙反弹，命中玩家后返还新的篮球次数
- `Rocket` 会高亮预计爆炸区域，并在命中后炸飞中心目标、推出周围目标
- `Teleport` 可以通过调试入口快速验证任意落脚点
- 头顶弧形 Tool 环会动态展示当前回合 Tool 列表
- 条件不满足的 Tool 会在 HUD 中置灰，点击后会弹出原因提示，同时客户端不会发送请求
- 定向 Tool 的预览与服务端结算复用同一套共享规则
- `Hookshot` 命中玩家时，客户端会在对方脚底下显示命中预览圈
- 加速带经过效果、幸运方块奖励和坑洞复活都由共享层统一裁定，客户端只负责渲染结果
- 多名玩家现在可以停留在同一格，客户端通过竖直叠放而不是平面错位来表现重叠

## 动画与文本回归

- 服务端现在会把最近一次动作的 `ActionPresentation` 序列化进房间状态
- 客户端 store 负责把它排入播放队列，并按本地 `simulationTimeMs` 推进
- `BoardScene.tsx` 会维护角色的“稳定落位”
  - 即使权威快照已经更新到最终坐标
  - 只要 motion 还没真正开始，场景仍保留角色上一稳定位置
  - 从而避免先瞬移到终点、再回起点播放动画
- 文本自动化现在优先依赖 `window.render_game_to_text()`
  - 除了原有的 `actionPresentation` 摘要外，现在还会输出 `displayedPlayers`
  - 回归脚本可以直接比较 `displayedPlayers` 与权威 `snapshot.players[].position`，确认动画交接是否正常
  - 当前动作表现链路
  - `Movement / Jump / Hookshot / Brake` 驱动角色位移动画
  - `Basketball / Rocket` 驱动投掷物飞行动画
  - `Rocket` 额外触发 `rocket_explosion` 特效事件

## 2026-03-30 Refactor Update

服务端房间层现在进一步拆成了：

- `WatcherRoom.ts`
  - 保留房间生命周期、回合推进、消息入口、共享规则调用
- `roomStateMappers.ts`
  - 把 Colyseus schema 转成共享层快照
- `roomStateMutations.ts`
  - 把共享层结算结果写回 schema
- `roomEventLog.ts`
  - 把共享层的 terrain / summon 触发结果转换成事件日志

这让服务端的职责更清晰：

- 共享层决定“规则结果是什么”
- 房间 helper 决定“如何把结果映射进 Colyseus schema”
- 房间类自身只负责“什么时候调用、调用后怎样推进 turn”

客户端内容配置也从组件里抽出一层：

- `packages/client/src/game/content/actionUi.ts`
  - 维护弧形 UI token、强调色、交互提示、方向箭头样式映射
- `packages/client/src/game/content/pets.ts`
  - 维护 cube-pets 模型 manifest、运行时路径和原始素材来源

最近又补了一层专门的场景资产目录：

- `packages/client/src/game/assets/board/`
  - 棋盘 tile 本体与地形装饰资产
- `packages/client/src/game/assets/previews/`
  - 落点圈、爆炸范围、建墙幽灵等预览资产
- `packages/client/src/game/assets/summons/`
  - 召唤物资产与 summon visual registry
- `packages/client/src/game/assets/presentation/`
  - 投射物与特效资产，以及按语义类型分发的 registry
- `packages/client/src/game/assets/player/`
  - 玩家脚下光环、当前回合标记等 presence 资产
- `packages/client/src/game/assets/shared/`
  - 资产共用的摆放辅助函数

`BoardScene.tsx` 现在主要负责：

- 读取快照与本地 aim 状态
- 生成预览 resolution
- 决定哪些资产需要被渲染

它不再直接内联定义大段程序化几何和动画 mesh，后续替换具体视觉时，更像是在换 asset，而不是改场景控制逻辑。

这样前端新增展示资源时，可以先改 manifest，再决定是否需要补新的渲染逻辑，而不是把路径或 token 直接硬编码进组件。
## 2026-03-31 Client Flow Follow-up

The client runtime flow is now split into smaller responsibilities:

- `App.tsx` only mounts the scene shell and global hooks.
- `HudSidebar.tsx` owns sidebar-only interaction and read-only player inspection UI.
- `useGameStore.ts` delegates playback progression to `state/presentationPlayback.ts`.
- Room command guards now delegate to `state/roomCommands.ts`.
- Shared selected-tool lookup and usability checks now live in `state/toolSelection.ts`.

On the scene side:

- aiming math is centralized in `interaction/aiming.ts`
- preview extraction is centralized in `interaction/previewState.ts`

This keeps the client preview path closer to the same shared authority input shape used by the room commands.

## 2026-03-31 Presentation Playback Follow-up

The scene playback path now distinguishes between authoritative state and displayed state more explicitly.

- `displayedPlayers`
  - built from active motion playback first
  - falls back to pending motion origins
  - finally falls back to authoritative player positions
- `displayedTiles`
  - built from the authoritative board plus pending `state_transition` rollbacks
- `displayedSummons`
  - built from the authoritative summon list plus pending `state_transition` rollbacks

This means the scene can now present these cases naturally:

- a mover finishes on the new authoritative cell without snapping back after playback
- an earth wall stays visible until the impact frame, then disappears
- a wallet stays visible until the mover actually reaches the pickup cell

Stacking also now follows cell entry order instead of player identity:

- later arrivals render on the bottom layer immediately
- existing occupants rise upward over a short stack animation
- `render_game_to_text()` exposes stack serial/index for text-only regression checks

## 2026-03-31 Golden Web Runner

The client now has a dedicated golden-case runner entry:

- route:
  - `/?mode=goldens`
  - `/goldens`
- component:
  - `packages/client/src/game/components/GoldenCaseRunnerApp.tsx`

This page does not connect to the room.
Instead, it imports the shared golden-case registry and runs cases locally in the browser one by one.

For each case it shows:

- step results
- assertion mismatches
- expected board text
- actual board text
- final summarized state

This makes the Web runner a debug and inspection surface, while the CLI runner remains the fastest batch check.

## 2026-04-01 Golden Playback Upgrade

The golden Web route is now closer to the real client scene instead of a summary-only dashboard.

- `GoldenCaseRunnerApp.tsx` now:
  - builds each case through the shared simulator
  - feeds `snapshot + presentation` into the local client store
  - reuses `GameBoardCanvas` and `BoardScene` for playback
- the page still does not connect to `WatcherRoom`
  - no websocket
  - no Colyseus room
  - no network shell in the test path

This gives the browser runner two roles at once:

- it is still a pass/fail inspector for golden assertions
- it is now also a visual playback surface for animation and timing review

## 2026-04-01 Scene Detail Follow-up

客户端场景交互这轮又补了几项更贴近正式体验的小机制。

### 长按说明卡

- `BoardScene.tsx` 现在支持对三类场景对象做长按检查：
  - 地形
  - 角色
  - 召唤物
- 长按后会在 3D 窗口顶部显示一张说明卡。
  - 卡片内容来自客户端内容配置和共享层注册信息
  - 松开鼠标后立即隐藏
- 当前主要用途：
  - 快速解释地形效果
  - 查看角色简介
  - 查看召唤物用途

### 弧形 Tool 环状态

- 场景头顶弧形 Tool 环现在不再在瞄准时卸载，而是切换为隐藏状态。
- 这样做有两个直接收益：
  - 瞄准时仍然满足“UI 不挡操作”的要求
  - 回合中新增 Tool 时，原地就可以播放插卡动画，而不是因为组件重建丢失状态

### Tool 选中与箭头显示

- `useGameStore.ts` 现在会在成功发送 `useTool` 指令后主动清空 `selectedToolInstanceId`
- `useWatcherConnection.ts` 也不再在快照同步时自动挑选下一个可用 Tool
- 结果是：
  - 用完 Tool 后不会偷偷切到下一张卡
  - 方向箭头也会随之消失
  - 左栏和场景的选中态保持一致

### 新工具插卡动画

- `SceneInteractionHud.tsx` 会跟踪 action phase 中新增的 `toolInstanceId`
- 当工具列表里出现新实例时，会给对应弧形按钮播放一次插入动画
- 当前已经覆盖这类来源：
  - 幸运方块奖励
  - 钱包奖励
  - 未来其他“中途派生新 Tool”的效果也可以直接复用

## 2026-04-01 Multiplayer Golden Follow-up

共享黄金案例现在额外覆盖了几条多人规则链路：
