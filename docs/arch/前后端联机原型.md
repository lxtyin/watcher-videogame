# 前后端联机原型

## 目标

联机层的职责是把 shared 规则接到真实房间与客户端表现上，而不是重新实现游戏规则。

## 服务端

### 关键文件

- `packages/server/src/rooms/WatcherRoom.ts`
- `packages/server/src/rooms/roomStateMappers.ts`
- `packages/server/src/rooms/roomStateMutations.ts`
- `packages/server/src/schema/WatcherState.ts`

### 当前职责

- 创建房间与地图状态
- 处理加入、离开、重连、准备、开局、回房、踢人
- 接收客户端高层命令
- 将 schema 映射到 shared snapshot/runtime
- 调用 shared orchestration
- 将 shared 结果回写到 schema
- 在需要时按 presentation 时长延迟调用 `advanceTurn()`

### 明确不负责

`WatcherRoom.ts` 不负责：

- 选择下一位玩家
- 写 `roll -> turn-action -> finishTurn -> nextTurn`
- 竞速终点排名推进
- 回合开始停留结算

这些逻辑都已经收口到 `packages/shared/src/gameOrchestration.ts`。

### 当前调用链

1. 房间收到消息，例如 `rollDice`、`useTool`。
2. `WatcherRoom` 用 `roomStateMappers.ts` 将 schema 转成 `GameSnapshot`。
3. `WatcherRoom` 读取本地保存的 `GameRuntimeState`。
4. `WatcherRoom` 调用：
   - `createGameOrchestrator(...).dispatch(command)`
   - 或 `createGameOrchestrator(...).advanceTurn()`
5. `roomStateMutations.ts` 把 shared 结果写回 schema。
6. 如果 shared 结果带有待推进状态，room 只按 presentation 时长挂一个 timer，稍后再次调用 `advanceTurn()`。

### 这条边界为什么重要

房间层现在只做“适配”和“调度”。这保证：

- golden、本地 simulator、真实房间使用同一份 turn flow
- room 改协议时不会顺手生出第二份规则
- 竞速终点这类时间敏感逻辑不再在 server 与 simulator 两边漂移

## 客户端

### 关键文件

- `packages/client/src/game/network/useWatcherConnection.ts`
- `packages/client/src/game/utils/deserializeRoomState.ts`
- `packages/client/src/game/components/BoardScene.tsx`
- `packages/client/src/game/interaction/`
- `packages/client/src/game/state/`

### 当前职责

- 建立 Colyseus 连接
- 将 schema 状态转为前端快照
- 提供本地输入、瞄准、预览与 HUD
- 消费 shared presentation 做动画播放
- 在 UI 层阻止明显非法输入，但不承担规则权威

### 表现层链路

1. server 同步最新 `WatcherState`。
2. client 反序列化为 `GameSnapshot`。
3. store 将 `latestPresentation` 放入本地播放队列。
4. `BoardScene` 基于 authoritative snapshot 与 displayed state 渲染：
   - 玩家位置
   - 地块变化
   - 召唤物变化
   - finish 动画退场

## schema 与 shared snapshot 的关系

schema 是网络同步格式，shared snapshot 是规则格式。

当前两者之间有专门的映射层：

- `roomStateMappers.ts`
  - schema -> shared
- `roomStateMutations.ts`
  - shared -> schema

当前玩家能力相关字段的边界是：

- schema 中同步 `tagsJson + modifiers`
- shared snapshot 中同步 `Player.tags + Player.modifiers`
- 角色静态能力仍由 shared 内容注册与技能系统推导

这样做的目的：

- Colyseus 结构变化不会直接污染 shared 规则
- shared 规则结构调整时，server 只需要改映射层
- room 类本身不再混合“规则 + 序列化 + 生命周期”三种职责

## 对未来 AI 的要求

如果你要改联机行为：

- 想改消息协议、重连、房间阶段、timer，可以改 `WatcherRoom.ts`
- 想改 schema 与 snapshot 的字段映射，可以改 mapper/mutation
- 想改真实结算、换人、终点推进、回合开始逻辑，必须回到 `packages/shared/src/gameOrchestration.ts`

不要在 `WatcherRoom.ts` 或 `simulation/engine.ts` 里补 turn logic 特判。
