# 共享规则层

## 位置

- `packages/shared/src/constants.ts`
- `packages/shared/src/types.ts`
- `packages/shared/src/board.ts`
- `packages/shared/src/dice.ts`
- `packages/shared/src/tools.ts`
- `packages/shared/src/actions.ts`
- `packages/shared/src/terrain.ts`

## 职责

共享层只保存纯数据和纯逻辑，不依赖浏览器或服务端运行时。
当前职责包括：

- 棋盘尺寸与默认地图配置
- 地块、玩家、回合、事件等共享类型
- 移动骰子与工具骰子的确定性掷骰逻辑
- Tool 定义注册表
- Tool 条件判断
- Tool 实例的消耗、派生与统一动作结算
- 地形行为注册与“经过 / 停留”结算

## 当前共享数据模型

当前共享类型的核心设计点如下：

- `ToolId`
  - 当前支持 `movement`、`jump`、`hookshot`、`dash`、`brake`、`buildWall`、`basketball`、`rocket`、`teleport`
- `TurnToolSnapshot`
  - 表示“本回合工具列表中的一个工具实例”
  - 关键字段包括 `instanceId`、`toolId`、`charges`、`params`
- `ToolDefinition`
  - 集中描述 Tool 名称、说明、目标模式、颜色、默认参数和条件
  - 当前目标模式支持 `instant`、`direction`、`tile`
  - 对 `tile` Tool 还可以进一步配置目标采样模式，例如 `axis_line`、`adjacent_ring`、`board_any`
- `ToolCondition`
  - 当前已落地一种基础条件：`tool_present`
- `ToolDieFaceDefinition`
  - 描述工具骰单个面会发放什么 Tool，以及该 Tool 的基础参数和次数
- `ActionResolution`
  - 统一返回角色位置变化、路径、预览高亮地块、地块变化、受影响玩家和新的 Tool 列表
- `TileDefinition`
  - 现在支持方向型地形元数据，例如 `conveyor.direction`
- `PlayerSnapshot`
  - 现在同步出生点与本回合地形标记，例如幸运方块的每回合限次标记
  - 玩家不再声明“占据格子”，同一坐标上可以存在多个玩家
- `TriggeredTerrainEffect`
  - 统一描述坑洞复活、幸运方块奖励与加速带效果，供服务端日志和客户端预览复用

这样做的目的，是让客户端预览、服务端权威判定、以及 UI 的可用性判断都复用同一套共享结构。

## 掷骰与 Tool 注册

`dice.ts` 提供确定性掷骰逻辑：

- `rollMovementDie(seed)`
  - 从移动骰面中得到本回合的 `Movement(points)`
- `rollToolDie(seed)`
  - 从工具骰面配置中得到本回合的基础 Tool

`tools.ts` 提供 Tool 注册表与 Tool 辅助逻辑：

- `TOOL_DIE_FACES`
  - 工具骰子可能出现的配置化 Tool 面
- `TOOL_DEFINITIONS`
  - Tool 元数据的单一来源
- `createToolInstance()`
  - 基于定义默认值和参数覆写生成统一 Tool 实例
- `createMovementToolInstance()`
  - 创建带点数的 `Movement` Tool
- `createRolledToolInstance()`
  - 根据骰面配置创建基础 Tool 实例
- `createDebugToolInstance()`
  - 创建调试入口发放的 Tool 实例
- `consumeToolInstance()`
  - 消耗一个 Tool 实例
- `getToolAvailability()`
  - 基于当前 `tools[]` 判断某个 Tool 是否满足使用条件

## 统一动作结算

`actions.ts` 是当前共享层最核心的入口。
它不再区分“Move 特判”和“Tool 结算”，而是统一处理“激活 Tool 实例之后会发生什么”。

`terrain.ts` 负责补上另一条扩展轴：地图格本身的行为。当前每个地形都可以独立声明：
- `onPassThrough`
  - 玩家在地面移动路径中经过该格时触发
- `onStop`
  - 一项 Tool 结算完成后，玩家最终停留在该格时触发

### 当前结算模式

- 输入
  - 当前棋盘
  - 当前玩家位置
  - 当前玩家 `tools[]`
  - 被激活的 Tool 实例
  - 可选方向或目标格
- 输出
  - 新的玩家位置
  - 新的 `tools[]`
  - 路径预览
  - 地块变化
  - 受影响玩家变化

### 当前 Tool 规则

- `Movement`
  - 沿指定方向持续前进，直到撞墙、越界、被阻挡或点数不足
  - 执行成功后整个 `Movement` 实例被消耗
- `Movement` / `Brake`
  - 复用同一条地面遍历逻辑，因此地面地形、墙体、土墙消耗和经过效果只需要维护一份
- `Push`
  - 现在直接复用与 `Movement` / `Brake` 相同的地面遍历逻辑
  - 这意味着推动也会像普通平移一样触发加速带、土墙消耗和其他经过效果
- `Jump`
  - 朝指定方向跳跃，最多跳到 `jumpDistance`
  - 作为飞跃型移动，不触发加速带这类经过效果
- `Hookshot`
  - 向前探测 `hookLength` 格，命中墙体时将自己拉向落点，命中玩家时尝试把对方拉近
  - 若目标格上叠有多个玩家，则会按共享规则一次结算这些命中目标
- `Dash`
  - 消耗自身，并给当前列表中的所有 `Movement` 增加 `dashBonus`
  - 只有当前列表里存在 `Movement` 时可用
- `Brake`
  - 根据拖拽选中的目标格，吸附到同轴方向
  - 朝该方向最多移动 `brakeRange` 格，若拖得更远则停在实际可达格
- `BuildWall`
  - 在周围八格中选择一格空地，生成耐久为 `wallDurability` 的土墙
- `Basketball`
  - 按 `projectileRange` 飞行，遇墙按 `projectileBounceCount` 反弹，命中玩家后推动 `projectilePushDistance`
- `Rocket`
  - 按 `projectileRange` 飞行，命中后在爆炸点结算
  - 中心目标按 `rocketBlastLeapDistance` 视为飞跃
  - 周围四格目标按 `rocketSplashPushDistance` 向外平移
- `Teleport`
  - 直接将行动玩家传送到任意可落脚地块
- 当前地形原型
  - `pit`
    - 只处理停留，结算后立即把玩家送回出生点
  - `lucky`
    - 只处理停留，为当前行动玩家额外投一次工具骰，并用回合标记限制每回合最多一次
  - `conveyor`
    - 只处理经过；顺方向经过时给当前移动预算 +2，否则把后续移动方向改成地形方向

## 扩展方式

当前 Tool 系统按“注册定义 + 统一实例 + 统一结算”的方式扩展，避免把逻辑散落到房间和客户端中。

新增 Tool 时，通常只需要同步修改这些位置：

1. 在 `types.ts` 扩展 `ToolId`
2. 在 `tools.ts` 增加 `TOOL_DEFINITIONS` 条目，并配置默认参数
3. 如需通过工具骰出现，在 `tools.ts` 增加到 `TOOL_DIE_FACES`
4. 在 `actions.ts` 增加执行器并注册到 `TOOL_EXECUTORS`
5. 如需新条件，在 `types.ts` 扩展 `ToolCondition`，并在 `tools.ts` 的条件判断中补充分支
6. 如需新的参数，在 `types.ts` 扩展 `ToolParameterId`，然后让 Tool 定义和执行器都读取该参数
7. 如需新交互模式，在 `types.ts` 扩展 `ToolTargetMode` 或目标采样模式，再让客户端针对该模式实现目标采样与预览

新增地形时，通常只需要同步修改这些位置：

1. 在 `types.ts` 扩展 `TileType` 与所需的地形元数据
2. 在 `board.ts` 或关卡数据中注册地形布局
3. 在 `terrain.ts` 为该地形补充 `onPassThrough` 或 `onStop`
4. 如需前端表现，在 `BoardScene.tsx` 为该地形补一个程序化或素材化渲染

## 设计意图

- “移动”被吸收到 Tool 体系中，后续新增获得、复制、强化、替换移动能力时不需要再改一套独立回合资源。
- Tool 是否可用由共享层统一判断，客户端只负责渲染可用或禁用状态。
- `ActionResolution` 直接返回新的 Tool 列表，服务端和客户端都不需要重复写 Tool 变更逻辑。
- 目标选择的形状也被吸收到共享接口中，客户端只负责采样“方向”或“格子”，具体实际落点仍由共享层统一裁定。
- 地形被拆成与 Tool 并行的扩展点后，未来新增陷阱、增益地块、传送、滑行等规则时，不需要再改每个 Tool 的主执行器。
- 玩家不再阻挡格子后，角色交互更多依赖 Tool 与地形结算，而不是把“是否有人站着”硬编码进基础移动规则。

## 动作表现时间线

共享层现在会把“权威结算结果”和“表现事件时间线”分开表达：

- `ActionResolution`
  - 继续负责最终位置、地块变化、受影响玩家和新的 `tools[]`
- `ActionPresentation`
  - 负责描述这次动作应该如何播放
  - 当前事件类型为 `player_motion`、`projectile`、`effect`
- `player_motion`
  - 使用路径点和 `motionStyle`
  - 当前支持 `ground` 与 `arc`
- `projectile`
  - 用于篮球、火箭等飞行物
  - 客户端根据 `projectileType` 决定模型和插值高度
- `effect`
  - 用于爆炸、脉冲、闪光等短时表现
  - 当前已落地 `rocket_explosion`

这层表达的意义是：

1. Tool 规则继续只关心语义结果，不把 mesh、缓动或粒子细节塞进共享规则
2. 服务端只需要同步语义化时间线，客户端可以独立选择具体动画和特效实现
3. 未来新增特效时，通常只需要扩展 `types.ts` 的表现事件枚举、在 `actions.ts` 追加事件，并在客户端渲染映射里补一种表现

## 2026-03-30 Refactor Update

本轮把共享层进一步拆成了“注册表 + 执行层”两部分。

### 内容注册表

`packages/shared/src/content/` 现在集中维护：

- `schema.ts`
- `tools.ts`
- `characters.ts`
- `summons.ts`
- `effects.ts`
- `defaultBoard.ts`

其中 `types.ts` 不再手写大段 id union，而是从注册表推导：

- `ToolId`
- `CharacterId`
- `SummonId`
- `PresentationEffectType`

这让新增内容时优先改注册表，再让执行层读取注册表，减少重复维护。

### 规则执行层

`packages/shared/src/rules/` 当前拆成：

- `spatial.ts`
  - 平移、飞跃、推动、投射物路径、爆炸预览等空间计算
- `toolExecutors.ts`
  - 各 Tool 的直接规则
- `actionResolution.ts`
  - 统一 blocked / applied 结果、Tool 消耗、停留后处理
- `actionPresentation.ts`
  - `player_motion / projectile / effect` 语义事件构造

`actions.ts` 现在只保留统一入口和少量公开 helper，不再承载所有实现细节。

### 当前扩展建议

新增 Tool 时，优先按下面顺序接入：

1. 在 `content/tools.ts` 注册定义与骰面
2. 在 `rules/toolExecutors.ts` 补执行器
3. 如需新表现事件，在 `content/effects.ts` 与 `actionPresentation.ts` 补语义描述

新增地图内容时：

1. 在 `content/defaultBoard.ts` 或未来的新地图内容文件中补布局
2. 在 `terrain.ts` 补地形行为
3. 客户端只消费结算结果和内容元数据，不重写规则判断

## 2026-03-31 Presentation State Transition Follow-up

Shared action resolution now treats visual-timing-sensitive board changes as first-class semantic events.

- `actionResolution.ts` appends `state_transition` events after the authoritative tool result is finalized.
- Each transition records:
  - the affected tile or summon
  - the `before` state
  - the `after` state
  - the semantic `startMs` when the visible change should happen
- Timing is aligned to motion arrival when possible.
  - A ground move can therefore break an earth wall exactly on impact.
  - A pass-through summon can disappear exactly when the mover reaches that cell.

This keeps the rule boundary clean:

- the shared layer decides *what changed* and *when it should become visible*
- the client decides *how that visible change is rendered*

That pattern should be reused for future cases such as:

- trap arming/disarming
- delayed destruction
- pickup reveal/hide
- hit flashes or area markers that should persist until an impact frame
- Movement-style Tools live in `rules/executors/movementTools.ts`.
- Projectile-style Tools live in `rules/executors/projectileTools.ts`.
- Board mutation / summon placement Tools live in `rules/executors/boardTools.ts`.

This keeps new Tools closer to "pick a group, add an executor, register it once" instead of extending one long switch-style file.

## 2026-03-31 Golden Runner

The shared layer now also owns a pure golden-case runner:

- `packages/shared/src/goldens/types.ts`
  - case DSL for scene setup, steps, and end-state expectations
- `packages/shared/src/goldens/layout.ts`
  - compact symbol-board parser and serializer for small test maps
- `packages/shared/src/goldens/runner.ts`
  - executes the same shared tool rules without depending on Colyseus or React
- `packages/shared/src/goldens/cases/`
  - registered golden-case fixtures

The key design choice is that golden cases do not call room code.
They execute directly through the shared rule entry and then compare a normalized end-state summary.

That keeps tests aligned with the architecture boundary:

- room code proves synchronization and orchestration
- golden cases prove the rule semantics themselves

## 2026-04-01 Movement Trigger Refactor

本轮把“位移”进一步整理成共享层的一等概念，目标是让地形、召唤物和未来更多内容都复用同一条触发链路。

### 位移语义

- 位移类型现在明确区分为：
  - `translate`
  - `leap`
  - `drag`
- 位移性质现在明确区分为：
  - `active`
  - `passive`
- Tool 可以通过 `actorMovement` 声明自己的行动者位移语义。
  - 当前 `Movement / Brake` 是主动平移
  - `Jump` 是主动飞跃
  - `Hookshot` 自身位移是主动拉拽
- 受影响玩家的位移也会显式记录。
  - 例如 `Basketball / Rocket splash` 是被动平移
  - `Rocket blast` 是被动飞跃
  - `Hookshot` 拉别人时是被动拉拽

### 统一触发阶段

共享层现在围绕显式的 movement record 驱动触发器，而不是再根据某个 Tool 单独特判。

- `onStart`
  - 位移开始时触发
- `onPass`
  - 每经过一格时触发
  - 最后一格也算经过
- `onStop`
  - 最终停留时触发

这意味着“经过”天然覆盖“最后一格”，而“停留”是一个额外的语义阶段。

### 地形与召唤物的边界

- 地形继续保留在 `terrain.ts`
  - 传送带这类会改变移动过程本身的规则，仍在 grounded traversal 中即时结算
  - 停留型地形继续在 action finalize 阶段统一处理
- 召唤物现在接入 `summons.ts` 的同一套 movement trigger
  - 不再单独维护一条“pass through summon”特例链路
  - 每个召唤物都可以独立声明 `onStart / onPass / onStop`

### 当前示例规则

- `conveyor`
  - 仅在 `translate` 的 `onPass` 中触发
  - 不区分 `active / passive`
- `wallet`
  - 在 `active translate` 或 `active drag` 的 `onPass` 中触发
  - 在 `active leap` 的 `onStop` 中触发
  - 因此被篮球、火箭等被动位移推到钱包上时不会拾取

### 设计收益

- Tool executor 现在只需要回答“谁以什么方式移动了、走过了哪些格子”。
- 地形和召唤物各自根据 movement descriptor 判断是否触发，不再反向依赖具体 Tool id。
- 未来如果加入新的召唤物、轨道、陷阱或角色被动，只要它们的判定能表达成“某种位移阶段触发”，就可以丝滑接入现有链路。

## 2026-04-02 Unified Movement System

本轮继续把旧逻辑收紧，目标是让所有 Tool 都只能调用同一套底层位移系统，而不是各自维护一份私有位移流程。

### 底层位移类型

- 共享层现在把底层位移固定为四种：
  - `translate`
  - `leap`
  - `drag`
  - `teleport`
- 运行时的 `MovementDescriptor` 现在不仅携带：
  - `type`
  - `disposition`
- 还会额外携带：
  - `timing`
    - `in_turn`
    - `out_of_turn`
  - `tags`
    - 供未来内容附加自由描述符，例如 `tool:hookshot`、`rocket:splash`

### 统一入口

- 新的底层位移入口在 `packages/shared/src/rules/movementSystem.ts`
- 当前对外使用的四个共享函数：
  - `resolveLinearDisplacement()`
  - `resolveLeapDisplacement()`
  - `resolveTeleportDisplacement()`
  - `resolveCurrentTileStop()`
- `Movement / Brake / Hookshot / Basketball / Rocket / Bomb Throw / Teleport`
  - 现在都通过这套入口完成真实位移
- Tool executor 仍然可以保留自己的“选目标”逻辑
  - 例如 `Hookshot` 的射线命中判定
  - 例如 `Rocket` 的爆炸中心选择
- 但命中后的位移、停留、地形触发、召唤物触发都不再自己重写

### 触发阶段

- 共享层现在只保留两类位移触发器：
  - `onPassThrough`
  - `onStop`
- 不再使用 `onStart`
- 语义约定更新为：
  - 最后一格也算 `onPassThrough`
  - `onStop` 是在最终停留点额外再结算一次
- 这意味着：
  - 地形和召唤物都能在位移过程中即时影响后续位移
  - 不需要再在 Tool 结束后拿整条 path 回放一遍 summon 逻辑

### 地形与召唤物

- `terrain.ts` 和 `summons.ts` 现在都走“即时 patch”模型
  - 允许在触发时修改后续方向
  - 允许在触发时修改剩余移动点数
  - 允许在触发时追加 Tool、消耗 summon、推进工具骰种子
- 当前落地规则：
  - `conveyor`
    - 只在 `translate` 的 `onPassThrough` 中响应
  - `wallet`
    - 只在主动平移或主动拖拽的 `onPassThrough` 中响应
    - 只在主动飞跃的 `onStop` 中响应
    - 不会响应被动位移
  - `pit / lucky`
    - 继续在 `onStop` 中响应

### 回合开始

- 回合开始时，当前玩家现在会立刻对脚下格子执行一次 `onStop`
- 这条逻辑由 `resolveCurrentTileStop()` 统一提供
- simulator 与 `WatcherRoom` 都已经接入这条入口
- 结果上：
  - 站在幸运方块上开始回合，会在掷骰前先拿到额外 Tool
  - 站在坑上开始回合，会在掷骰前先回出生点
- 因此旧的 `onStart` 概念已经不再需要

### 设计收益

- 新增 Tool 时，优先只需要决定：
  - 如何选目标
  - 触发哪一种底层位移
  - 位移 descriptor 带什么标签
- 新增 summon / terrain 时，优先只需要决定：
  - 响应 `onPassThrough` 还是 `onStop`
  - 响应哪些 movement descriptor
  - 是否修改后续位移过程
- 这让“控制手段”和“地图/召唤物反馈”真正解耦，后续继续扩内容时不会再回到每个 Tool 单独写一份位移规则的旧模型。

## 2026-04-02 Gameplay Mode Layer

本轮在 shared 层补上了“地图绑定玩法模式”的最小结构，避免竞速规则散落在 server / client 两边。

### 地图注册

- 新增 `packages/shared/src/content/maps.ts`
- 当前注册两张地图：
  - `free_default`
  - `race_sprint`
- 每张地图定义现在会一起声明：
  - `mode`
  - `label`
  - `allowDebugTools`
  - `layout`
  - `spawnMode`
  - `spawnPositions`

### Snapshot 扩展

`GameSnapshot` 和 `PlayerSnapshot` 现在额外带上：

- `mapId`
- `mapLabel`
- `mode`
- `allowDebugTools`
- `settlementState`
- `players[].finishRank`
- `players[].finishedTurnNumber`

这样客户端不需要再从事件日志反推“谁已经完赛”。

### 终点判定

- 棋盘新增：
  - `start`
  - `goal`
- `goal` 通过 terrain `onStop` 触发共享语义事件：
  - `TriggeredTerrainEffect.kind = "goal"`
- 真正的竞速结算不写在 terrain 本身，而是由更高层根据：
  - 当前 `mode`
  - 当前 turn number
  - `goal` 触发结果
  来更新玩家名次和结算状态

### 当前收益

- 地形层只负责表达“玩家在终点停下了”
- simulator / room 负责表达“这在 race 模式里意味着完赛”
- free 模式即使以后复用 `goal` 地块，也不会被迫带上竞速规则
