# 共享规则层

## 目标

`packages/shared` 是 Watcher 的唯一规则权威层。只要某个机制会影响真实对局结果，它就必须先在这里实现，再由 server 和 client 复用。

## 当前模块划分

### 回合编排

- `packages/shared/src/gameOrchestration.ts`
  - 统一处理高层命令与 turn flow。
  - 当前命令入口包括：
    - `rollDice`
    - `useTool`
    - `endTurn`
    - `setCharacter`
    - `grantDebugTool`
    - `advanceTurn`
  - 同时维护：
    - 阶段切换
    - 骰子与工具实例分配
    - 事件日志
    - presentation sequence
    - race 结算与延迟推进

### 动作结算

- `packages/shared/src/actions.ts`
  - 统一工具动作入口。
  - 负责把单个工具动作收敛成 `ActionResolution`。
- `packages/shared/src/rules/`
  - `movementSystem.ts`
    - 底层移动结算，只负责位移解析本身。
  - `tool-modules/`
    - 按工具拆分的定义与执行模块。
  - `actionResolution.ts`
    - 通用动作后处理，负责整理真实结算结果、表现序列与状态切换时机。
  - `actionPresentation.ts`
    - 语义表现事件。
  - `previewDescriptor.ts`
    - 统一封装预览描述。
- `packages/shared/src/toolInteraction.ts`
  - 统一定义 `Tool.interaction`、`ToolSelectionRecord`、基础交互阶段工厂与选择值读写辅助。

### 交互输入契约

- `ToolContentDefinition` 不再声明 `targetMode` 或 `tileTargeting`。
- 每个工具模块必须在自己的文件中同时定义：
  - `definition`
  - `interaction`
  - `execute`
- 交互顺序由 `Tool.interaction` 描述，合法范围与预览内容由工具自己的规则逻辑决定。
- `UseToolCommandPayload` 的权威结构固定为：
  - `{ toolInstanceId, input: ToolSelectionRecord }`
- `ToolSelectionRecord` 是命名 slot 到选择值的映射，当前基础值类型只有：
  - `direction`
  - `tile`
  - `choice`
- shared 工具执行时只能读取 `context.input`，不得再依赖平铺的 `direction / targetPosition / choiceId` 字段。
- 新工具如需多段输入，应在 `Tool.interaction` 中组合阶段，而不是向 room、simulator 或 client store 新增工具专属 payload 字段。

### 能力系统

- `packages/shared/src/skills/`
  - 角色能力模块，按技能拆分。
- `packages/shared/src/modifiers.ts`
  - `Skill`、`Modifier`、hook 上下文、运行时插拔辅助与返回值接口。
- `packages/shared/src/playerTags.ts`
  - `Player.tags` 读写辅助。
- `packages/shared/src/characters.ts`
  - 角色定义物化与查询。

### 内容注册

- `packages/shared/src/content/`
  - `tools.ts`
  - `characters.ts`
  - `summons.ts`
  - `maps.ts`
  - `effects.ts`
  - `boards/defaultBoard.ts`

### 地形与召唤物

- `packages/shared/src/terrain.ts`
  - 地形 `onPassThrough / onStop`。
- `packages/shared/src/summons.ts`
  - 召唤物 `onPassThrough / onStop`。

## 为什么回合编排必须收口到 shared

以下逻辑只允许出现在 `gameOrchestration.ts`：

- 当前玩家是否合法行动
- `turn-start -> turn-action -> turn-end` 的阶段推进
- 回合开始与结束的停留结算
- `resolveCurrentTileStop` 对应的停留结算编排
- `turn-action` 结束后的继续行动、结束回合或延迟推进
- race 模式完赛后的名次与结算
- “等待 finish 动画播放完再继续”的待推进状态

原因有三点：

1. room 和 simulator 过去维护过两套 turn flow，已经产生过语义漂移。
2. golden、房间权威和客户端本地回放都需要复用同一份高层时序。
3. 只要回合编排分叉，新增工具、技能、地形或召唤物时就很容易只修一边。

## 明确禁止

未来 AI 不应在以下文件中编写回合逻辑：

- `packages/server/src/rooms/WatcherRoom.ts`
- `packages/shared/src/simulation/engine.ts`

这两个位置只允许：

- 组织输入输出
- 调用 `dispatch(command)` / `advanceTurn()`
- 做状态映射
- 做计时器调度

## 工具、地形、召唤物、能力的接入位置

### 新增工具

1. 在 `packages/shared/src/content/tools.ts` 注册静态定义。
2. 在 `packages/shared/src/tool-modules/` 实现定义与执行逻辑。
3. 如涉及角色专属主动能力，再在 `packages/shared/src/skills/` 中接入发放逻辑。
4. 如有前端专属表现，再去 client 增补 UI 与视觉映射。

### 新增地形

1. 在地图内容中放置 tile。
2. 在 `packages/shared/src/terrain.ts` 注册触发逻辑。
3. 如有视觉差异，再去 client 做表现映射。

### 新增召唤物

1. 在 `packages/shared/src/content/summons.ts` 注册定义。
2. 在 `packages/shared/src/summons.ts` 编写触发逻辑。
3. 在 client 的 summon visual registry 中补表现。

### 新增角色能力

1. 主动能力优先建模为 `Tool`。
2. 被动能力优先建模为 `Skill -> Modifier`。
3. 数值或布尔状态统一进入 `Player.tags`，运行时安装的 Modifier 统一进入 `Player.modifiers`。
4. 如现有 hook 不够用，优先扩展通用 hook，而不是新增角色特判。
5. 先阅读 [能力系统统一模型](./能力系统统一模型.md) 再改代码。

## 表现层边界

shared 只表达：

- 状态变化
- 事件日志
- `PreviewDescriptor`
- `ActionPresentation`

client 负责：

- 预览结果的视觉映射
- PlaybackEngine 的时间推进与瞬态采样
- Mesh / 材质 / 动画实现
- HUD、提示与输入体验

shared 不关心具体 mesh，client 不重写真实规则。

### PreviewDescriptor

`PreviewDescriptor` 是 shared 规则层给表现层的预览描述结构。

它当前统一包含：

- 所有玩家的预计落点
- 当前玩家的移动路径
- 选择范围
- 作用范围
- 当前预览是否合法

约束：

- shared 只负责给出这些语义字段。
- 每个工具如何把这些字段画成圈、轨迹、虚影、投射物辅助线，由 client 自己决定。
- 未来 AI 不应再把某个工具的预览素材或 mesh 细节塞回 shared。

### ActionPresentation

`ActionPresentation` 是 authoritative 动作已经发生后的语义事件序列。

当前只允许三种事件：

- `motion`
  - 玩家或投射物的运动。
- `reaction`
  - 纯表现反应，不改逻辑状态。
- `state_transition`
  - 真正切换显示状态。

约束：

- 如果某个动画只是表现反馈，不应伪装成规则状态变化。
- 如果某个表现需要延后切图、延后切可见性，应该表达成 `state_transition` 的时机，而不是让 client 自己猜。
- 如果未来新增表现类型，优先扩展这三类事件的载荷，而不是恢复零散动画特判。

## 对 AI 的实现清单

修改 shared 规则前，先检查：

1. 这是不是会影响真实结算结果。
2. 如果是，它是不是应该进入 `gameOrchestration.ts`、`actions.ts`、`terrain.ts`、`summons.ts`，或能力系统定义的 `Tool / Modifier` 扩展点。
3. room 和 simulator 是否仍然只是调用方，而不是第二份规则实现。
4. 新状态是否正确落在 `Player.tags`，以及新插拔能力是否正确落在 `Player.modifiers`，而不是又新建并行容器。
5. 新预览是否应该进入 `PreviewDescriptor`，新动画是否应该进入 `ActionPresentation` 的 `motion / reaction / state_transition` 三类事件，而不是散落到 client 特判里。
