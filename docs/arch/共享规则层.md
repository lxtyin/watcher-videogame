# 共享规则层

## 目标

`packages/shared` 是 Watcher 的唯一规则权威层。只要某个机制会影响真实对局结果，它就应该先在这里落地，再由 server 和 client 复用。

## 当前模块划分

### 回合编排

- `packages/shared/src/gameOrchestration.ts`
  - 负责高层命令与回合流转。
  - 处理：
    - `rollDice`
    - `useTurnStartAction`
    - `useTool`
    - `endTurn`
    - `setCharacter`
    - `grantDebugTool`
    - `advanceTurn`
  - 同时维护：
    - 事件日志
    - 骰子种子
    - tool instance 序号
    - presentation sequence
    - 竞速完赛后的待推进状态

### 动作结算

- `packages/shared/src/actions.ts`
  - 统一工具动作入口。
- `packages/shared/src/rules/`
  - `movementSystem.ts`
    - 底层移动系统。
  - `executors/`
    - 各类工具执行器。
  - `actionResolution.ts`
    - 通用后处理。
  - `actionPresentation.ts`
    - 语义表现事件。

### 内容注册

- `packages/shared/src/content/`
  - `tools.ts`
  - `characters.ts`
  - `summons.ts`
  - `maps.ts`
  - `effects.ts`
  - `boards/defaultBoard.ts`

### 角色运行时

- `packages/shared/src/characterRuntime.ts`
  - 角色状态的跨回合记忆。
  - 回合开始动作。
  - 角色回合负载工具注入。
  - 回合结束清理。

### 地形与召唤物

- `packages/shared/src/terrain.ts`
  - 地形 `onPassThrough / onStop`。
- `packages/shared/src/summons.ts`
  - 召唤物 `onPassThrough / onStop`。

## 为什么回合编排必须收口到 shared

这次重构后，以下逻辑都只允许出现在 `gameOrchestration.ts`：

- 当前玩家是否合法行动
- 从 `roll` 进入 `action`
- 回合结束后的换人
- 回合开始停留结算
- 竞速玩家到达终点后的排名与结算推进
- “等待 finish 动画播完再继续”的待推进状态

原因有三点：

1. room 和 simulator 过去各自维护一套 turn flow，已经出现语义漂移。
2. golden、房间权威、客户端本地回放都需要同一份高层时序。
3. 一旦回合编排分叉，未来新增角色技能、地形或召唤物时，很容易只修一边。

## 明确禁止

未来的 AI 不应在以下文件中编写回合逻辑：

- `packages/server/src/rooms/WatcherRoom.ts`
- `packages/shared/src/simulation/engine.ts`

上面两个位置只能：

- 组织输入输出
- 调用 `dispatch(command)` / `advanceTurn()`
- 做状态映射
- 做计时器调度

如果需要新增回合分支、结算条件、推进状态，请修改 `packages/shared/src/gameOrchestration.ts`，而不是在 room 或 simulator 补一个特判。

## 工具、地形、召唤物、角色的接入位置

### 新增工具

1. 在 `packages/shared/src/content/tools.ts` 注册定义。
2. 在 `packages/shared/src/rules/executors/` 添加执行器。
3. 如需新表现事件，在 `content/effects.ts` 与 `actionPresentation.ts` 补语义。

### 新增地形

1. 在地图内容中放置 tile。
2. 在 `packages/shared/src/terrain.ts` 注册触发逻辑。
3. 如果有客户端专属视觉，再去 client 做表现映射。

### 新增召唤物

1. 在 `packages/shared/src/content/summons.ts` 注册。
2. 在 `packages/shared/src/summons.ts` 编写触发逻辑。
3. 在 client 的 summon visual registry 中补表现。

### 新增角色能力

1. 在 `packages/shared/src/content/characters.ts` 注册角色静态定义。
2. 在 `packages/shared/src/characterRuntime.ts` 接入跨回合状态与回合资源。
3. 如有角色专属工具，仍按工具扩展路径接入。

## 表现层边界

shared 只表达：

- 状态变化
- 事件日志
- `ActionPresentation`

client 负责：

- Mesh/材质/动画实现
- 时间推进与 displayed state 回滚
- UI 提示与瞄准体验

shared 不关心具体 mesh；client 不重写规则。

## 对 AI 的实现清单

当你准备改规则时，先检查：

1. 这是不是会影响真实结算结果？
2. 如果是，它是否应该进入 `packages/shared/src/gameOrchestration.ts`、`actions.ts`、`terrain.ts`、`summons.ts` 或 `characterRuntime.ts`？
3. room 或 simulator 是否只是调用方，而不是第二份规则实现？

只要第 3 点的答案是否定的，就不应该继续在 room/simulator 中动手。
