# 共享规则层

## 目标

`packages/shared` 是 Watcher 的唯一规则权威层。
只要某个机制会影响真实对局结果，它就必须先在这里实现，再由 server 和 client 复用。

## 当前模块划分

### 回合编排

- `packages/shared/src/gameOrchestration.ts`
  - 统一处理高层命令与 `turn-start -> turn-action -> turn-end` 阶段推进
  - 负责阶段入口停留结算、事件日志、presentation sequence 与竞速结算

### 动作结算

- `packages/shared/src/actions.ts`
  - 统一工具动作入口
- `packages/shared/src/rules/actionDraft.ts`
  - 定义 `ToolActionDraft`，作为 shared 主链的唯一结算草稿
- `packages/shared/src/rules/movementSystem.ts`
  - 统一底层位移结算；所有真实位移都必须经过这里
  - 对外的位移 resolver 接收 `MovementDescriptor`，直接消费调用方确定的位移类型、主动/被动与标签
  - `translate` 被 solid tile 以剩余步数阻挡时，会在这里 author 通用 `impact` 语义与撞墙反弹表现
- `packages/shared/src/rules/previewDescriptor.ts`
  - 统一构造 `PreviewDescriptor`
- `packages/shared/src/rules/actionPresentation.ts`
  - 统一构造 `ActionPresentation`
- `packages/shared/src/rules/actionResolution.ts`
  - 基于 tile / summon mutation 追加 `state_transition`

### 展示文本

- `getToolTextDescription(tool)`
  - 调用工具模块自己的 `getTextDescription`
  - 工具展示文本由工具模块手写；shared 不再用统一参数标签表自动拼接描述
- `getTerrainTextDescription(tile)`
  - 调用地形模块自己的 `getTextDescription`
- `getSkillTextDescription(skillId)`
  - 调用技能定义自己的 `getTextDescription`

shared 只提供文字语义和数值，不规定具体 UI 排版；client 不应为工具、地形或技能再维护一份含义描述。

### 能力系统

- `packages/shared/src/tool-modules/`
  - 按工具拆分的定义、`isAvailable`、preview 与 execute
- `packages/shared/src/toolInteraction.ts`
  - 工具交互协议与输入结构
- `packages/shared/src/skills/`
  - 按技能拆分的 `Skill -> Modifier`
- `packages/shared/src/modifiers.ts`
  - modifier hook 定义
- `packages/shared/src/playerTags.ts`
  - `Player.tags` 读写辅助

### 地形与召唤物

- `packages/shared/src/terrain-modules/`
  - 每个地形一个文件
  - 地形只能通过接收并修改 draft 的方式干预主流程
  - 当前统一入口包括 `onPassThrough / onStop / onImpact`
- `packages/shared/src/terrain.ts`
  - 地形分发入口
- `packages/shared/src/summons/`
  - 每个召唤物一个文件，`index.ts` 负责统一分发
  - 召唤物当前统一入口包括 `onPassThrough / onStop`

### 触发系统

- 地形、召唤物、Skill都有其触发条件
- `onPassThrough` 在位移路径上触发，通常不包括终点
- `onStop` 在一段位移的终点触发；进入 `turn-action` 时，`gameOrchestration.ts` 也会通过 `applyPhaseEntryStop()` 对当前格子统一触发一次地形与召唤物的 `onStop`
- `turn-start` 不触发当前格子的 `onStop`；它只处理回合开始 hook、空幸运方块恢复、眩晕跳过等阶段入口规则
- `Skill` 在游戏的各个阶段已经注册了多种Hook，并且未来可以继续增加

### 内容注册

- `packages/shared/src/tool-modules/index.ts`
- `packages/shared/src/content/characters.ts`
- `packages/shared/src/content/summons.ts`
- `packages/shared/src/content/effects.ts`
- `packages/shared/src/content/boards/defaultBoard.ts`

## 地形框架

当前地形系统已经收口成“每种地形一个模块”的模型。

- `floor`
  - 空白地块
- `wall`
  - 阻挡地面移动与投射物
- `earth-wall`
  - 阻挡投射物；被撞碎时通过 presentation 播放碎裂特效
- `boxing-ball`
  - 阻挡地面移动与投射物
  - 被任意 impact 命中时播放摇摆 effect；被当前回合玩家以 `translate` 撞击时奖励一个同等力度的拳击
- `highwall`
  - 阻挡地面移动、飞跃穿越与投射物
- `start`
  - 出生点
- `goal`
  - 终点
- `conveyor`
  - 停留时沿朝向移动
- `poison`
  - 停留时触发，把玩家送回出生点
  - 使用玩家 `motion(style = "fall_side")` 表达中毒侧倒
- `pit`
  - 经过时触发，把玩家送回出生点并中断本次移动
  - 使用玩家 `motion(style = "spin_drop")` 表达旋转坠落
- `cannon`
  - 停留时触发，直接复用火箭核心结算
- `lucky`
  - 停留时给予工具，并把地形从 `lucky` 切换为 `emptyLucky`
  - 通过 `reaction + state_transition` 表达领取动画与显隐切换
- `emptyLucky`
  - 空幸运方块
  - 在下一位玩家回合开始时统一恢复为 `lucky`

## Draft 规则

shared 主链已经统一成“自顶向下传递 draft”的结算模式。

- 上层负责决定调用顺序
- 下层 resolver、tool executor、terrain trigger、summon trigger 只负责读取并修改 draft
- 不再允许返回一组 patch 交给上层二次合并

这意味着：

- `Tool.execute()` 应直接修改 `ToolActionDraft`
- 地形与召唤物触发器应直接修改同一个 draft
- 嵌套触发产生的 presentation 应直接追加到同一个 draft 中

## 关键边界

### 回合边界

以下逻辑只允许写在 `packages/shared/src/gameOrchestration.ts`：

- 阶段推进
- 阶段入口、回合结束与当前格停留结算
- finish 动画等待与竞速结算

以下文件不得再编写回合逻辑：

- `packages/server/src/rooms/WatcherRoom.ts`
- `packages/shared/src/simulation/engine.ts`

### 位移边界

所有真实位移都必须经过 `packages/shared/src/rules/movementSystem.ts`。
任何 Tool、Modifier、Terrain 或 Summon 都不应直接修改玩家位置。

`resolveLinearDisplacement`、`resolveDragDisplacement`、`resolveLeapDisplacement` 与 `resolveTeleportDisplacement` 负责消费最终 `MovementDescriptor` 并产出路径、触发器、副作用与表现时间线。
`MovementDescriptor` 由调用方在进入位移系统前确定，包含 `type / disposition / tags` 三类规则语义。
`timing` 不属于 descriptor；地形、召唤物与技能触发器需要判断当前位移是否发生在玩家自己的回合时，通过发起动作的 `actorId` 与被移动玩家 id 计算。
工具模块通过 `resolveToolMovementDescriptor()` 显式生成主动或被动工具位移；工具定义元数据不承载默认位移类型。

`resolveLeapDisplacement` 的中间格按飞跃处理；最后落点按 `landing` 语义处理，因此经过触发型地形或召唤物在落点可以按落点语义触发。

### 表现边界

shared 只负责语义层：

- 最终状态变化
- `PreviewDescriptor`
- `ActionPresentation`
- 事件日志

shared 不负责 mesh、材质、shader 或客户端播放细节。

### 地形边界

地形模块只处理地形自身的触发与展示文本。
地面移动、飞跃穿越、投射物阻挡与落点合法性属于底层位移和空间规则，集中在 `movementSystem.ts` 与 `spatial.ts`。
位移系统 author 出的通用 `impact` 语义会再分发给地形模块；地形自身不负责重新判断“这次算不算撞击”。

未来 AI 不应：

- 在 `gameOrchestration.ts` 中为单个地形加专用变量
- 在 room、simulator 或 client 中重写地形阻挡或触发结算

## 扩展入口

### 新增工具

1. 在 `packages/shared/src/tool-modules/` 增加工具模块
2. 在工具定义内提供 `interaction / isAvailable / getTextDescription`
3. 在 `packages/shared/src/tool-modules/index.ts` 接入模块、执行器与骰面
4. 通过 draft 完成 preview 与 execute

### 新增地形

1. 在 `packages/shared/src/content/schema.ts` 扩展 tile 类型
2. 在 `packages/shared/src/terrain-modules/` 增加单文件模块
3. 如需新的底层阻挡或落点语义，更新 `movementSystem.ts` 与 `spatial.ts`
4. 如需前端表现，再接入 client 的 board 资源与 presentation 资源

### 新增召唤物

1. 在 `packages/shared/src/content/summons.ts` 注册定义
2. 在 `packages/shared/src/summons/` 增加召唤物模块，并在 `index.ts` 接入触发逻辑

## 对未来 AI 的要求

修改 shared 规则前，先检查：

1. 这是否影响真实结算结果
2. 如果影响，是否应该进入 shared 主链
3. 新逻辑是否在修改 draft，而不是返回 patch
4. 是否把地形、工具、技能的专用规则收进各自模块，而不是写进 orchestrator、room 或 scene
## 起床战争补充

- `bedwars` 是 shared 正式支持的第三种 `GameMode`
- 阵营信息不通过复制 tile type 承载，而是统一收敛为：
  - `Player.teamId`
  - `TileDefinition.faction`
- 起床战争相关胜负、回合推进、死亡后是否复活，统一写在 `gameOrchestration.ts` 与 `gameplay.ts`，不允许回退到 room 层特判
- `pit / poison` 等致死地形只表达“玩家被击倒”，是否复活由 shared 主链根据当前模式和塔是否存活裁决
- 塔、阵营出生点、阵营营地都作为普通 terrain module 接入 `onStop / onImpact`，不额外开平行规则系统

### 阵营地形

- `tower`
  - `solid tile`
  - 仅敌方角色撞击时掉耐久
  - 击碎后切为 `floor`
- `teamSpawn`
  - 仅承载阵营出生语义
  - bedwars 下玩家复活点从这里推导
- `teamCamp`
  - 仅己方角色在自己回合停留时奖励一次随机工具骰

### 眩晕

- `basis:stun` 是通用 runtime modifier，不属于某个角色
- 它通过 `ModifierPhaseHookResult.skipTurn` 在 `turn-start` 要求直接跳过回合，并在同一时刻清除自己
