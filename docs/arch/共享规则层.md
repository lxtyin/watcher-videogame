# 共享规则层

## 目标

`packages/shared` 是 Watcher 的唯一规则权威层。只要某个机制会影响真实对局结果，它就必须先在这里实现，再由 server 和 client 复用。

## 当前模块划分

### 回合编排

- `packages/shared/src/gameOrchestration.ts`
  - 统一处理高层命令与 turn flow
  - 负责阶段推进、骰子分配、事件日志、presentation sequence、race 结算与延迟推进
  - `resolveCurrentTileStop` 的停留结算编排也已收口到这里

### 动作结算

- `packages/shared/src/actions.ts`
  - 统一工具动作入口
- `packages/shared/src/rules/actionDraft.ts`
  - 统一的 `ResolutionDraft / ToolActionDraft`，负责在 shared 主链中承载实时状态、表现事件与结算结果
- `packages/shared/src/rules/movementSystem.ts`
  - 底层位移结算；输入 draft 并直接修改 draft
- `packages/shared/src/tool-modules/`
  - 按工具拆分的定义与执行模块；executor 只允许修改 draft，不再返回 patch
- `packages/shared/src/rules/actionResolution.ts`
  - 只保留输入读取与 state transition presentation 辅助，不再承载旧的 applied/blocked patch builder
- `packages/shared/src/rules/previewDescriptor.ts`
  - 统一构造 `PreviewDescriptor`
- `packages/shared/src/rules/actionPresentation.ts`
  - 统一构造 `ActionPresentation`

### 交互与能力

- `packages/shared/src/toolInteraction.ts`
  - `Tool.interaction`、`ToolSelectionRecord` 与基础交互阶段工厂
- `packages/shared/src/skills/`
  - 技能模块
- `packages/shared/src/modifiers.ts`
  - skill / modifier 接口与 hook 上下文
- `packages/shared/src/playerTags.ts`
  - `Player.tags` 读写辅助
- `packages/shared/src/characters.ts`
  - 角色物化与查询

### 内容注册

- `packages/shared/src/content/`
  - `tools.ts`
  - `characters.ts`
  - `summons.ts`
  - `maps.ts`
  - `effects.ts`
  - `boards/defaultBoard.ts`

### 地形与召唤物

- `packages/shared/src/terrain.ts`
  - 地形触发；只接收 draft 与局部状态并直接修改 draft
- `packages/shared/src/summons.ts`
  - 召唤物触发；只接收 draft 与局部状态并直接修改 draft

## 关键边界

### 回合边界

以下逻辑只允许写在 `gameOrchestration.ts`：

- `turn-start -> turn-action -> turn-end` 的阶段推进
- 回合开始与结束的停留结算
- `resolveCurrentTileStop` 对应的停留结算编排
- 延迟推进、finish 动画等待与 race 结算

不得在以下文件中编写回合逻辑：

- `packages/server/src/rooms/WatcherRoom.ts`
- `packages/shared/src/simulation/engine.ts`

### 移动边界

所有真实位移都必须走：

- `packages/shared/src/rules/movementSystem.ts`

任何 `Tool` 或 `Modifier` 都不应直接修改玩家位置。
`movementSystem` 不再向上返回成组 patch；它只负责修改 draft，并返回本次位移路径与局部表现信息。

### 表现边界

shared 只表达语义结果：

- 最终状态变化
- 事件日志
- `PreviewDescriptor`
- `ActionPresentation`

Preview 与 presentation 的权威定义见 [表现层原型](./表现层原型.md)。

### 能力边界

主动能力统一建模为 `Tool`，被动能力统一建模为 `Skill -> Modifier`。权威定义见 [能力系统统一模型](./能力系统统一模型.md)。

## 扩展入口

### 新增工具

1. 在 `packages/shared/src/content/tools.ts` 注册静态定义。
2. 在 `packages/shared/src/tool-modules/` 实现模块。
3. 模块内部只能通过 `ToolActionDraft` 修改结算状态；不要恢复 `buildAppliedResolution / buildBlockedResolution` 一类 patch builder。
3. 如涉及角色专属发放逻辑，再到 `packages/shared/src/skills/` 接入。

### 新增地形

1. 在地图内容中放置 tile。
2. 在 `packages/shared/src/terrain.ts` 注册触发逻辑。
3. 地形触发器必须直接修改 draft；不要返回 `nextPosition / nextTools / tileMutations` 之类 patch。

### 新增召唤物

1. 在 `packages/shared/src/content/summons.ts` 注册定义。
2. 在 `packages/shared/src/summons.ts` 编写触发逻辑。
3. 召唤物触发器必须直接修改 draft；不要恢复旧的 phase resolution patch 模型。

### 新增角色能力

1. 主动能力优先建模为 `Tool`。
2. 被动能力优先建模为 `Skill -> Modifier`。
3. 状态统一进入 `Player.tags + Player.modifiers`。
4. 如果现有 hook 不够用，优先扩展通用 hook。

## 对未来 AI 的要求

修改 shared 规则前，先检查：

1. 这是不是会影响真实结算结果。
2. 如果是，它是不是应该进入 `gameOrchestration.ts`、`actions.ts`、`terrain.ts`、`summons.ts`，或能力系统的 `Tool / Modifier` 扩展点。
3. room 和 simulator 是否仍然只是调用方，而不是第二份规则实现。
4. 新状态是否正确落在 `Player.tags + Player.modifiers`。
5. 新预览或新动画是否应该进入 shared 语义结构，而不是散落到 client 特判里。
