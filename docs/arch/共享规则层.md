# 共享规则层

## 位置

- `packages/shared/src/constants.ts`
- `packages/shared/src/types.ts`
- `packages/shared/src/board.ts`
- `packages/shared/src/dice.ts`
- `packages/shared/src/tools.ts`
- `packages/shared/src/actions.ts`
- `packages/shared/src/terrain.ts`

## 职责

共享层只保存纯数据和纯逻辑，不依赖浏览器或服务端运行时。
当前职责包括：

- 棋盘尺寸与默认地图配置
- 地块、玩家、回合、事件等共享类型
- 移动骰子与工具骰子的确定性掷骰逻辑
- Tool 定义注册表
- Tool 条件判断
- Tool 实例的消耗、派生与统一动作结算
- 地形行为注册与“经过 / 停留”结算

## 当前共享数据模型

当前共享类型的核心设计点如下：

- `ToolId`
  - 当前支持 `movement`、`jump`、`hookshot`、`dash`、`brake`、`buildWall`、`basketball`、`rocket`、`teleport`
- `TurnToolSnapshot`
  - 表示“本回合工具列表中的一个工具实例”
  - 关键字段包括 `instanceId`、`toolId`、`charges`、`params`
- `ToolDefinition`
  - 集中描述 Tool 名称、说明、目标模式、颜色、默认参数和条件
  - 当前目标模式支持 `instant`、`direction`、`tile`
  - 对 `tile` Tool 还可以进一步配置目标采样模式，例如 `axis_line`、`adjacent_ring`、`board_any`
- `ToolCondition`
  - 当前已落地一种基础条件：`tool_present`
- `ToolDieFaceDefinition`
  - 描述工具骰单个面会发放什么 Tool，以及该 Tool 的基础参数和次数
- `ActionResolution`
  - 统一返回角色位置变化、路径、预览高亮地块、地块变化、受影响玩家和新的 Tool 列表
- `TileDefinition`
  - 现在支持方向型地形元数据，例如 `conveyor.direction`
- `PlayerSnapshot`
  - 现在同步出生点与本回合地形标记，例如幸运方块的每回合限次标记
  - 玩家不再声明“占据格子”，同一坐标上可以存在多个玩家
- `TriggeredTerrainEffect`
  - 统一描述坑洞复活、幸运方块奖励与加速带效果，供服务端日志和客户端预览复用

这样做的目的，是让客户端预览、服务端权威判定、以及 UI 的可用性判断都复用同一套共享结构。

## 掷骰与 Tool 注册

`dice.ts` 提供确定性掷骰逻辑：

- `rollMovementDie(seed)`
  - 从移动骰面中得到本回合的 `Movement(points)`
- `rollToolDie(seed)`
  - 从工具骰面配置中得到本回合的基础 Tool

`tools.ts` 提供 Tool 注册表与 Tool 辅助逻辑：

- `TOOL_DIE_FACES`
  - 工具骰子可能出现的配置化 Tool 面
- `TOOL_DEFINITIONS`
  - Tool 元数据的单一来源
- `createToolInstance()`
  - 基于定义默认值和参数覆写生成统一 Tool 实例
- `createMovementToolInstance()`
  - 创建带点数的 `Movement` Tool
- `createRolledToolInstance()`
  - 根据骰面配置创建基础 Tool 实例
- `createDebugToolInstance()`
  - 创建调试入口发放的 Tool 实例
- `consumeToolInstance()`
  - 消耗一个 Tool 实例
- `getToolAvailability()`
  - 基于当前 `tools[]` 判断某个 Tool 是否满足使用条件

## 统一动作结算

`actions.ts` 是当前共享层最核心的入口。
它不再区分“Move 特判”和“Tool 结算”，而是统一处理“激活 Tool 实例之后会发生什么”。

`terrain.ts` 负责补上另一条扩展轴：地图格本身的行为。当前每个地形都可以独立声明：
- `onPassThrough`
  - 玩家在地面移动路径中经过该格时触发
- `onStop`
  - 一项 Tool 结算完成后，玩家最终停留在该格时触发

### 当前结算模式

- 输入
  - 当前棋盘
  - 当前玩家位置
  - 当前玩家 `tools[]`
  - 被激活的 Tool 实例
  - 可选方向或目标格
- 输出
  - 新的玩家位置
  - 新的 `tools[]`
  - 路径预览
  - 地块变化
  - 受影响玩家变化

### 当前 Tool 规则

- `Movement`
  - 沿指定方向持续前进，直到撞墙、越界、被阻挡或点数不足
  - 执行成功后整个 `Movement` 实例被消耗
- `Movement` / `Brake`
  - 复用同一条地面遍历逻辑，因此地面地形、墙体、土墙消耗和经过效果只需要维护一份
- `Push`
  - 现在直接复用与 `Movement` / `Brake` 相同的地面遍历逻辑
  - 这意味着推动也会像普通平移一样触发加速带、土墙消耗和其他经过效果
- `Jump`
  - 朝指定方向跳跃，最多跳到 `jumpDistance`
  - 作为飞跃型移动，不触发加速带这类经过效果
- `Hookshot`
  - 向前探测 `hookLength` 格，命中墙体时将自己拉向落点，命中玩家时尝试把对方拉近
  - 若目标格上叠有多个玩家，则会按共享规则一次结算这些命中目标
- `Dash`
  - 消耗自身，并给当前列表中的所有 `Movement` 增加 `dashBonus`
  - 只有当前列表里存在 `Movement` 时可用
- `Brake`
  - 根据拖拽选中的目标格，吸附到同轴方向
  - 朝该方向最多移动 `brakeRange` 格，若拖得更远则停在实际可达格
- `BuildWall`
  - 在周围八格中选择一格空地，生成耐久为 `wallDurability` 的土墙
- `Basketball`
  - 按 `projectileRange` 飞行，遇墙按 `projectileBounceCount` 反弹，命中玩家后推动 `projectilePushDistance`
- `Rocket`
  - 按 `projectileRange` 飞行，命中后在爆炸点结算
  - 中心目标按 `rocketBlastLeapDistance` 视为飞跃
  - 周围四格目标按 `rocketSplashPushDistance` 向外平移
- `Teleport`
  - 直接将行动玩家传送到任意可落脚地块
- 当前地形原型
  - `pit`
    - 只处理停留，结算后立即把玩家送回出生点
  - `lucky`
    - 只处理停留，为当前行动玩家额外投一次工具骰，并用回合标记限制每回合最多一次
  - `conveyor`
    - 只处理经过；顺方向经过时给当前移动预算 +2，否则把后续移动方向改成地形方向

## 扩展方式

当前 Tool 系统按“注册定义 + 统一实例 + 统一结算”的方式扩展，避免把逻辑散落到房间和客户端中。

新增 Tool 时，通常只需要同步修改这些位置：

1. 在 `types.ts` 扩展 `ToolId`
2. 在 `tools.ts` 增加 `TOOL_DEFINITIONS` 条目，并配置默认参数
3. 如需通过工具骰出现，在 `tools.ts` 增加到 `TOOL_DIE_FACES`
4. 在 `actions.ts` 增加执行器并注册到 `TOOL_EXECUTORS`
5. 如需新条件，在 `types.ts` 扩展 `ToolCondition`，并在 `tools.ts` 的条件判断中补充分支
6. 如需新的参数，在 `types.ts` 扩展 `ToolParameterId`，然后让 Tool 定义和执行器都读取该参数
7. 如需新交互模式，在 `types.ts` 扩展 `ToolTargetMode` 或目标采样模式，再让客户端针对该模式实现目标采样与预览

新增地形时，通常只需要同步修改这些位置：

1. 在 `types.ts` 扩展 `TileType` 与所需的地形元数据
2. 在 `board.ts` 或关卡数据中注册地形布局
3. 在 `terrain.ts` 为该地形补充 `onPassThrough` 或 `onStop`
4. 如需前端表现，在 `BoardScene.tsx` 为该地形补一个程序化或素材化渲染

## 设计意图

- “移动”被吸收到 Tool 体系中，后续新增获得、复制、强化、替换移动能力时不需要再改一套独立回合资源。
- Tool 是否可用由共享层统一判断，客户端只负责渲染可用或禁用状态。
- `ActionResolution` 直接返回新的 Tool 列表，服务端和客户端都不需要重复写 Tool 变更逻辑。
- 目标选择的形状也被吸收到共享接口中，客户端只负责采样“方向”或“格子”，具体实际落点仍由共享层统一裁定。
- 地形被拆成与 Tool 并行的扩展点后，未来新增陷阱、增益地块、传送、滑行等规则时，不需要再改每个 Tool 的主执行器。
- 玩家不再阻挡格子后，角色交互更多依赖 Tool 与地形结算，而不是把“是否有人站着”硬编码进基础移动规则。
