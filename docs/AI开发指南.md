# AI开发规约

## 文档阅读和维护

1. 执行开发任务前，先阅读 `docs/index.md`，按需继续阅读相关架构文档。
2. 完成任务后，必须同步维护文档，维护文档严格遵从下面的规则：
   1. 先将本轮执行的任务内沉淀到 `progress.md`。
   2. 仅当涉及架构修改、核心概念重构时，才修改 `index.md` 以及 `arch` 下的架构文档。
   6. `游戏规则与内容定义.md` 是用户制定的规则文档，包含了概念定义和现有角色/工具的期望功能介绍，你可以阅读但**不能修改**。
3. 架构文档撰写约束（针对arch下所有文档）：
   1. **永远呈现当前状态**：架构文档应当描述当前代码架构、数据流、关键入口文件和核心机制的定义。
   2. **不关注历史信息**：架构文档不应提及已修复的 bug、废弃设计或迁移过程，不能使用任何“已经修复”、“变更为”等隐含历史状态的措辞。
   3. **仅概念，无内容**：架构文档可以举例，但不应该罗列具体的游戏内容，不要在其中详尽地介绍每一个工具/角色
4. 文档始终使用中文编写。

## 代码规范

1. 遵循现有代码风格和命名规范，核心功能添加注释。
2. 不要在主逻辑中加入适配某个Tool或Skill的专用参数和特判。
3. 终极目标：低耦合、高内聚、开闭原则、强可扩展性。

## 测试

1. 测试以系统能正常运行、关键输出正确为准，不要进行截图/图像识别测试。
2. 优先使用 `npm.cmd run goldens` 进行黄金案例测试。
3. 执行性能相关任务前后，参照 `run-heavy-golden-benchmark.mjs` 进行性能测试并对比分析
