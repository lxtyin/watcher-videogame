# 《Watcher》 游戏电子化架构设计文档

## 1. 整体技术选型

为了实现高扩展性、低耦合度以及稳定的多人联机体验，本项目采用 **权威服务器（Authoritative Server）** 模式，前后端使用统一的语言以最大化代码复用。

| **模块**     | **技术栈**                | **说明**                                                     |
| ------------ | ------------------------- | ------------------------------------------------------------ |
| **核心语言** | `TypeScript`              | 提供严格的类型检查，是保证低耦合、高扩展和AI代码生成质量的基础。 |
| **前端框架** | `React` + `Zustand`       | `React` 负责UI组件树，`Zustand` 负责轻量级的本地状态管理。   |
| **3D 渲染**  | `React Three Fiber (R3F)` | 声明式的 3D 渲染库，将地图网格、角色模型封装为 React 组件，易于维护。 |
| **后端框架** | `Node.js` + `Colyseus`    | `Colyseus` 专为回合制/实时联机设计，自带房间管理和状态增量（Delta）同步功能。 |
| **共享逻辑** | `Shared Package`          | 独立的代码包，包含寻路、碰撞检测、类型定义，前后端共同引入。 |

------

## 2. 系统层级划分

系统严格分为三层，确保表现（画面）与逻辑（规则）分离：

### 2.1 表现层 (Client / Frontend)

- **职责**：只负责“画图”和“发指令”。
- **行为**：监听服务端的 `GameState` 变化，驱动 R3F 更新角色位置和特效；将玩家的鼠标点击转化为操作指令（如 `MoveCommand`, `UseItemCommand`）发送给服务端。
- **原则**：绝对不在此层计算游戏核心逻辑（如判断能否穿墙）。

### 2.2 权威逻辑层 (Server / Backend)

- **职责**：游戏的“上帝”，掌握唯一真实的状态。
- **行为**：接收客户端指令，验证合法性，调用规则引擎计算结果，更新房间状态，广播给所有玩家。

### 2.3 共享数据与规则层 (Shared)

- **职责**：存放前后端共用的接口（Interfaces）和纯函数（Pure Functions）。
- **包含内容**：
  - 常量配置（地图大小、骰子面数）。
  - 数据结构定义（`IPlayer`, `ITile`, `IItem`）。
  - 纯逻辑算法（`calculatePath(start, end, map)`）。

## 3. 核心机制设计 (高扩展性的关键)

为了满足未来“大量角色、复杂机制、回合外插入操作”的需求，我们采用以下三大核心设计模式：

### 3.1 基于命令模式的结算系统 (Command & Event Stack)

不直接修改数据，而是将所有行为封装为**命令**，压入**结算栈**中按顺序执行。这是实现“打断”、“反击”、“回合外插入”的基础。

- **流程示例**：
  1. 玩家 A 发起 `RocketAttackCommand(目标B)`。
  2. 指令入栈。系统广播 `OnBeforeAttack` 事件。
  3. 玩家 B 的被动技能触发，生成 `ShieldCommand`，**插入到栈顶**。
  4. 栈顶弹出结算：B 获得护盾。
  5. 次栈顶弹出结算：火箭爆炸，计算伤害/击退，因为有护盾，免除负面效果。

### 3.2 修饰器系统 (Modifier System)

放弃复杂的类继承，使用修饰器（Buff/Debuff）来管理所有的数值变化和特殊状态。

- **结构**：一个 Modifier 包含 `类型`、`数值`、`持续回合/时间`、`来源`。
- **应用场景**：
  - **派欧尼亚的被动**：开局挂载一个永不失效的 `Modifier(Type: DiceMoveBonus, Value: 2)`。
  - **冲刺道具**：使用后挂载一个持续1回合的 `Modifier(Type: DiceMoveBonus, Value: 2)`。
  - 在计算玩家最终移动力时，只需 `BaseMove + Sum(DiceMoveBonus Modifiers)`。新增一百个角色也只需配置不同的 Modifier，核心计算逻辑一行都不用改。

### 3.3 数据驱动的地图网格 (Data-Driven Grid)

地图由基于坐标的独立区块（Tile）组成。

- **结构**：二维坐标系 $(x, y)$ 作为唯一索引。
- **Tile 属性**：包含 `type`（如空地、墙、土墙）和 `state`（如土墙剩余强度 $X$）。
- **交互逻辑解耦**：当玩家移动经过 $(x, y)$ 时，触发该 Tile 的 `OnEnter` 钩子。如果是土墙，则在此钩子中处理扣除移动力和破坏墙体的逻辑。

## 4. 游戏核心状态数据结构 (GameState)

服务端维护的全局状态根节点，由 Colyseus 自动同步给客户端。

- `players`: `Map<String, Player>` (所有玩家的状态集合，包含坐标、血量、手牌/工具、当前修饰器)
- `board`: `Map<String, Tile>` (地图网格状态，键名为坐标如 "1,2")
- `turnInfo`: `Object` (当前回合信息，包括当前行动玩家ID、当前阶段、剩余移动点数)
- `eventLog`: `Array` (事件日志，用于前端播放动画和特效)

------

## 5. 性能与后续优化策略

- **视野与按需加载**：前端 3D 资产（角色模型、特效）使用 React 懒加载（`Suspense`），只加载当前对局中存在的角色资源。
- **增量同步 (Delta Sync)**：Colyseus 默认支持。每帧只下发发生改变的数据（如某面土墙耐久度下降），而不是整个地图，极大节省网络带宽。
- **对象池 (Object Pooling)**：对于频繁生成的 3D 物体（如火箭弹模型、篮球模型、爆炸粒子），在前端使用对象池复用，避免频繁触发垃圾回收（GC）导致游戏卡顿。
