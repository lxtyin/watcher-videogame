你指出的问题是对的：现在很多 resolver 本质上都在走“返回 patch，由上层合并”的模式。像 [terrain.ts](/E:/Develop/Watcher/packages/shared/src/terrain.ts)、[movementSystem.ts](/E:/Develop/Watcher/packages/shared/src/rules/movementSystem.ts)、[gameOrchestration.ts](/E:/Develop/Watcher/packages/shared/src/gameOrchestration.ts) 之间，确实有不少“下层产出修改，上层负责接”的样板代码。这个模式在机制少的时候还行，但一旦地形开始能主动发射火箭、连锁改地图、附带 presentation，就会越来越笨重。

我支持你改成`自顶向下传递 draft state / resolution`，但我会建议你把它收束成下面这个版本：

**推荐形态**
- 不要把完整 `GameSnapshot` 直接裸传给所有下层。
- 改成传一个`动作级草稿对象`，例如：
  - `draft.board`
  - `draft.players`
  - `draft.summons`
  - `draft.tools`
  - `draft.toolDieSeed`
  - `draft.presentation`
  - `draft.logs`
  - `draft.triggeredTerrainEffects`
  - `draft.triggeredSummonEffects`
- 所有 resolver / trigger 都拿这个 draft 来读写。
- 最外层只负责两件事：
  - 决定调用顺序
  - 在最后把 draft 收束成 `ActionResolution`

也就是说，`上层仍然控制时序，下层直接修改动作草稿`。这样比“所有层都能乱改全局 snapshot”稳很多。

**为什么我觉得这套更好**
- `resolveLinearDisplacement()` 不再需要层层返回 `tileMutations / summonMutations / nextTags / nextTools` 再合并。
- `onStop` / `onPassThrough` 可以直接改玩家位置、改地图、加动画。
- `cannon` 可以在地形触发里直接调用一个抽出来的 `resolveRocketCore(draft, spec)`，不需要再把一堆结果拆出来重新拼。
- Presentation 也更自然，直接往同一个 `draft.presentation.events` 里追加即可。

**但有一个边界必须守住**
这不意味着“下层可以接管主流程”。  
主流程仍然必须决定：
- 什么时候跑 `onPassThrough`
- 什么时候跑 `onStop`
- 什么时候结束移动
- 什么时候进入下一阶段
- 什么时候处理 race/finish

也就是：
- `orchestration` 决定`何时调用`
- `resolver/trigger` 决定`调用后怎么改 draft`

如果把“调用顺序”也下放，系统会很快失控。

**我建议的核心抽象**
可以做成这样：

```ts
interface ResolutionDraft {
  actorId: string;
  board: DraftBoardState;
  players: DraftPlayerState[];
  summons: DraftSummonState[];
  tools: TurnToolSnapshot[];
  toolDieSeed: number;
  presentation: ActionPresentationEvent[];
  eventLog: DraftEventLogEntry[];
  terrainEffects: TriggeredTerrainEffect[];
  summonEffects: TriggeredSummonEffect[];
  stopMovement: boolean;
}
```

然后所有函数改成这种味道：

```ts
resolveLinearDisplacement(draft, options): void
resolveStopTerrainEffect(draft, context): void
resolvePassThroughTerrainEffect(draft, context): void
resolveRocketCore(draft, spec): void
```

- [movementSystem.ts](/E:/Develop/Watcher/packages/shared/src/rules/movementSystem.ts)

**我最认同你的点**
是这句：`大炮地形也能直接调一个 resolveRocketTool 解决问题`。  
这正说明当前“工具结算”和“环境触发”还没有共享统一执行内核。把它们统一到 draft-based resolver 后，这个问题就顺了。

**我唯一不建议的点**
是“让触发器直接改完整地图快照”。  
我建议改的是：
- `动作级 draft`
- 不是 `整局游戏 snapshot`

因为：
- 预览也要复用这套逻辑
- 完整 snapshot 太大，职责太多
- 很容易让 terrain/skill 顺手去碰 turnInfo、pendingAdvance、settlement 之类本不该碰的东西

- 这会比现在的 patch-return 模式更适合地形系统扩展。
- 但应当落成`action-scoped mutable draft`，而不是“所有下层直接改全局游戏状态”。
